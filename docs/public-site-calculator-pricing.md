# Публичный сайт: калькулятор, прайсы и AI-ценообразование

Документ фиксирует, как должен работать публичный калькулятор сайта `stroyka26.pro`.

## Как считает сейчас

Калькулятор публичного сайта сейчас считает предварительную стоимость внутри фронтенда:

- базовые ставки и коэффициенты лежат в `src/components/PublicSitePage.jsx`;
- используются типы работ: дом, ремонт, коммерция, реконструкция;
- сумма считается по площади, типу объекта, уровню работ, материалам, срокам и коэффициентам;
- итог отправляется в CRM через `/site/leads` как снимок заявки;
- реальные сметы, прайс-листы ERP и рыночные цены пока не участвуют в расчете.

Это годится как рекламный ориентир, но не как управляемый коммерческий прайс.

## Как должно быть

Калькулятор должен брать цены не из hardcode, а из управляемой структуры:

1. Директор или замдиректора задает базовые ставки сайта.
2. Система хранит ставки по типам объектов, пакетам работ, регионам и коэффициентам.
3. AI анализирует рынок и предлагает изменения цен.
4. Директор утверждает или отклоняет предложения AI.
5. Только утвержденные цены попадают в публичный калькулятор.

AI не должен сам менять публичный прайс без подтверждения руководителя.

## Источники цены

Правильная цепочка источников:

1. Базовый прайс компании.
2. Реальные сметы и выполненные объекты из ERP.
3. Фактические закупки материалов.
4. Фактические выплаты мастерам и субподрядчикам.
5. Рыночные ориентиры, которые AI собирает как справочную аналитику.
6. Ручная корректировка директора.

## Таблицы, которые нужно добавить

Минимальная структура:

- `site_price_rules` — действующие правила калькулятора;
- `site_price_suggestions` — предложения AI по изменению цен;
- `site_market_snapshots` — сохраненные рыночные наблюдения;
- `site_gallery_projects` или расширение публичных полей объекта — галерея сайта.

## Логика AI-анализа рынка

AI должен работать в безопасном режиме:

- собирает рыночные ориентиры по категориям: дом, ремонт, коммерция, реконструкция;
- сравнивает с нашим текущим прайсом;
- объясняет причину изменения: материалы, зарплаты, сезонность, сложность, город;
- предлагает новую ставку и диапазон;
- ставит статус `На проверке`;
- директор утверждает ставку.

Пример результата:

```json
{
  "category": "repair",
  "segment": "capital",
  "region": "Ставропольский край",
  "currentRate": 32000,
  "suggestedRate": 35000,
  "confidence": 0.72,
  "reason": "Рост стоимости материалов и работ по отделке, текущая ставка ниже рынка"
}
```

## Доменная схема

Рекомендуемая схема:

- `stroyka26.pro` — публичный сайт, калькулятор, галерея, заявки;
- `app.stroyka26.pro` — рабочая ERP для сотрудников и клиентов;
- `api.stroyka26.pro` — можно добавить позже, если потребуется отдельный API-домен.

На старте можно оставить ERP на текущем домене, но правильнее развести публичный сайт и рабочий кабинет.

## Галерея проектов

В галерею нельзя просто вставлять картинки из Pinterest как готовые работы компании.

Правильные варианты:

- свои реальные фото объектов;
- фото с разрешением заказчика;
- лицензированные стоковые изображения;
- AI-сгенерированные визуализации с пометкой `пример`;
- Pinterest использовать только как референсы для дизайна, не как доказательство выполненных работ.

Каждый проект в галерее должен иметь:

- тип объекта;
- площадь;
- город;
- срок;
- диапазон стоимости;
- что сделано;
- фото;
- кнопку заявки.

## Связь с ERP

Уже есть:

- `/site/projects` — публично показывает объекты, отмеченные для сайта;
- `/site/pricing` — публично отдает ставки калькулятора;
- `/site/leads` — принимает заявки с сайта и заводит лид в CRM.
- `/site-price-rules` — закрытый endpoint директора для редактирования ставок.
- публичный сайт разделяет `Объекты`, `Наши работы` и `Референсы`, чтобы не смешивать реальные доказательства выполнения и будущие визуальные направления;
- партнерские заявки поставщиков, мастеров, бригад и субподрядчиков отправляются в CRM как заявки на проверку без автоматической выдачи доступа.

Нужно доработать:

- публикацию объектов в галерею с нормальными фото и описаниями;
- AI-предложения по ставкам;
- историю изменений публичных ставок;
- раздельную посадку: публичный сайт и ERP-кабинет.

## Этапы внедрения

1. ✅ Вынести ставки калькулятора из `PublicSitePage.jsx` в backend endpoint `/site/pricing`.
2. ✅ Добавить таблицу `site_price_rules`.
3. ✅ Добавить кабинет директора для редактирования ставок сайта.
4. ✅ Подключить калькулятор к `/site/pricing`.
5. ✅ Проверить nginx proxy: `/site/pricing`, `/site/projects`, `/site/leads`, `/site-price-rules` должны проксироваться в backend, а не отдавать `index.html`.
6. Добавить AI-предложения по ставкам без автоприменения.
7. ✅ Сделать галерею реальных/примерных проектов с фильтрами, отдельным блоком `Наши работы` и референсами по направлениям.
8. Развести публичный сайт и ERP по доменам.
9. Прогнать мобильную версию, SEO, заявки и CRM.
