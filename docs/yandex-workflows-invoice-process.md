# Yandex Workflow: invoice to warehouse process

Цель процесса: принять одно или несколько фото накладной, распознать документ,
собрать одну накладную, проверить НДС и материалы, вернуть в СтройКу готовую
структуру для сохранения на склад или объектный склад.

## Что автоматизируем

```text
Фото накладной
-> OCR / YandexGPT Vision
-> объединение страниц
-> структурированный JSON
-> проверка маршрута: склад / объект / мои траты / ручная проверка
-> проверка НДС и итогов
-> сопоставление материалов со сметой
-> сохранение в СтройКу или возврат needs_review
```

## Первый безопасный вариант

На первом этапе Workflow не пишет напрямую в базу. Он только возвращает
готовый результат в приложение, а пользователь нажимает "Сохранить".

Это защищает от ошибок распознавания и от неверного прихода на склад.

## Шаги настройки в Yandex Cloud

1. Создать сервисный аккаунт для Workflow.
2. Выдать ему доступ только к нужным сервисам: YandexGPT/Vision, Lockbox, вызов HTTP.
3. Сохранить секреты в Lockbox:
   - `STROYKA_API_URL`
   - `STROYKA_WORKFLOW_TOKEN` — то же значение, что `WORKFLOW_TOKEN` в `backend/.env`
   - `YANDEX_FOLDER_ID`
4. Создать Workflow `stroyka-invoice-to-warehouse`.
5. Добавить HTTP endpoint для запуска процесса.
6. В СтройКе добавить вызов Workflow из формы "Принять на склад".
7. На первом этапе результат Workflow показывать в форме, без автосохранения.
8. После стабилизации разрешить автосохранение только при `status=ok` и высокой уверенности.

## Логика шагов Workflow

### 1. Validate input

- Проверить, что есть хотя бы одно фото.
- Проверить, что выбран склад или объект.
- Проверить роль пользователя: директор, зам_директора, кладовщик, снабженец, прораб.

### 2. OCR pages

- Каждое фото отправить на распознавание.
- Сохранить текст каждой страницы отдельно.
- Не пытаться считать суммы до объединения страниц.

### 3. Merge document

- Объединить страницы в одну накладную.
- Убрать дубли заголовков и итогов.
- Сохранить `pagesDetected` и ссылки на все фото.

### 4. Text to JSON

- Передать объединенный текст в дообученную YandexGPT-модель.
- Требовать только валидный JSON.
- Если JSON невалидный, вернуть `needs_review`.

### 5. Estimate material match

- По каждому материалу найти кандидат в смете объекта.
- Использовать алиасы и нормализацию:
  - размеры;
  - бренд;
  - тип материала;
  - единица измерения;
  - пакет работ.
- Если уверенность ниже 0.75, не сохранять автоматически.

### 6. VAT and totals check

- Проверить:
  - сумма строк;
  - НДС;
  - итог с НДС;
  - итог без НДС.
- Если расхождение больше допустимого порога, вернуть `needs_review`.

### 7. Return result

- Вернуть готовую форму в приложение.
- Фото обязательно прикрепить к накладной.
- Пользователь видит результат и нажимает "Сохранить".

## Что должен делать backend СтройКи

На первом этапе достаточно добавить endpoint-обертку:

```text
POST /workflow/invoice/preview
```

Он принимает фото/параметры и возвращает безопасный предпросмотр в модальное
окно накладной. На первом этапе endpoint ничего не пишет в склад и не создает
трату.

Для шага `invoice-preview` в Yandex Workflow:

- имя шага: `invoice-preview` (без `_`, только латиница/цифры/дефисы);
- метод: `POST`;
- URL: `https://stroyka26.pro/workflow/invoice/preview`;
- заголовки:
  - `Content-Type: application/json`;
  - `X-Workflow-Token: <значение STROYKA_WORKFLOW_TOKEN>`;
- тело для первого теста:

```json
{
  "projectName": "Кисловодск Лицей 4",
  "warehouseTarget": "object",
  "selectedAction": "receive_to_warehouse",
  "photos": [],
  "user": {
    "role": "директор",
    "name": "Workflow test"
  }
}
```

Ожидаемый первый результат: HTTP 200, `route=warehouse_invoice`,
`status=needs_review`, предупреждение что фото не переданы. Это нормально:
мы проверяем связку Workflow -> backend без записи в базу.

Сохранение остается через существующий endpoint:

```text
POST /warehouse-invoices
```

## Правила безопасности

- Workflow не удаляет данные.
- Workflow не архивирует объекты.
- Workflow не пишет в склад, если `status=needs_review`.
- Workflow не записывает строительные материалы в "Мои траты", если пользователь
выбрал "Принять на склад".
- Все фото накладной должны оставаться прикрепленными к документу.

## Что тестировать

1. Одностраничная накладная без НДС.
2. Одностраничная накладная с НДС.
3. Накладная на 2-4 фото.
4. Накладная с материалом, который называется иначе, чем в смете.
5. Накладная с материалом не из сметы.
6. Ошибка маршрута: пользователь выбрал склад, ИИ не должен отправить в "Мои траты".
7. Плохое фото: должен быть `needs_review`, а не мусор в складе.
