# Единое внедрение "Карты руководителя"

Этот чеклист нужен, чтобы сначала собрать карту без привязки, посмотреть ее в песочнице, а потом внедрить разом и не сломать текущие цепочки Stroyka.

## До внедрения

- Утвержден экран `docs/prototypes/director-map/index.html`.
- Утвержден контракт `docs/prototypes/director-map/contract.v1.json`.
- Подтверждено, что карта не заменяет главную страницу и открывается внутри карточки объекта.
- Подтверждено, что красный сигнал не блокирует работу, склад, ЖПР, оплату или документы.
- Подтверждено, что фактические деньги считаются только из `project_payments`.
- Подтверждено, что сомнительные связи уходят в `needs_review`.
- Подтверждено, что первый live-режим будет read-only и под feature flag.

## Что нельзя делать в первом внедрении

- Проводить складские движения из карты.
- Подтверждать ЖПР из карты.
- Создавать фактические оплаты из карты.
- Автоматически переносить сроки этапов.
- Автоматически связывать документы с уверенностью ниже 90%.
- Удалять, схлопывать или нормализовать старые документы без ручной проверки.
- Считать счет, накладную и оплату как три расхода.

## Цепочки, которые нужно проверить

1. `объект -> этап -> пакет работ -> ответственный`
2. `смета -> норма/объем -> потребность материала`
3. `потребность -> заявка -> КП -> счет`
4. `счет -> отгрузка -> приемка -> претензия при недостаче`
5. `приемка -> накладная -> бухгалтерская проверка`
6. `накладная -> входной контроль -> сертификаты/фото`
7. `приход -> склад объекта -> выдача -> списание`
8. `выдача/работа -> ЖПР -> подтверждение факта`
9. `скрытая работа -> АОСР -> подписи/фото/печать`
10. `счет/накладная -> обязательство -> project_payments как факт денег`
11. `роль пользователя -> доступный объект -> доступный пакет работ`
12. `сигнал карты -> задача/переход -> audit_log`

## Порядок единого внедрения

### 1. Подготовить изолированный модуль

Будущая структура:

```text
src/features/director-map/
  ProjectDirectorMapPanel.jsx
  DirectorMapSummary.jsx
  DirectorMapTimeline.jsx
  DirectorMapStageCard.jsx
  DirectorMapSupplyChain.jsx
  DirectorMapLinksTable.jsx
  directorMapAdapter.js
  directorMapRules.js
```

До подключения к приложению модуль питается только моковым контрактом.

### 2. Сделать backend read-only агрегатор

Endpoint будущего внедрения:

```http
GET /projects/director-map?projectName=...
```

На первом live-шаге endpoint только читает данные. Он не создает связи, не пишет статусы и не меняет учет.

### 3. Включить feature flag

Пример флага:

```text
FEATURE_DIRECTOR_MAP=true
```

Если флаг выключен, карта полностью исчезает из UI и не влияет на объект.

### 4. Подключить UI одной точкой

Место:

- карточка объекта;
- группа `Объект`;
- вкладка `Карта руководителя`;
- на главной только компактный вход/сигнал, без замены главной.

### 5. Добавить связи только после read-only проверки

Таблица `project_work_links` добавляется не в первый шаг, а после проверки read-only данных.

Правило:

- уверенность 90% и выше - можно предложить как `auto`;
- ниже 90% - только `needs_review`;
- пользователь может подтвердить или отклонить связь;
- отклоненная связь не должна появляться снова как уверенная без нового основания.

### 6. Добавить зависимости только после связей

`project_work_relations` появляется после `project_work_links`.

На первом этапе зависимости только показываются как сигналы. Автоперенос сроков запрещен.

## Проверки перед включением

Команды для локальной проверки:

```bash
PYTHONPYCACHEPREFIX=/tmp/stroyka-pycache python3 -m py_compile backend/main.py
npm run build
npm run check:smeta
npm run smoke:supply-chain
npm run smoke:work-doc-chain
npm run smoke:role-matrix
npm run smoke:core-chains
```

Если часть smoke-команд требует окружение или доступы, это фиксируется как blocker перед внедрением, а не пропускается молча.

## Стоп-условия

Внедрение останавливаем, если:

- карта дает разные суммы денег по сравнению с финансами объекта;
- карта закрывает дефицит материала без объекта или пакета работ;
- карта считает неподтвержденный ЖПР как факт;
- роль видит чужой объект или чужой пакет;
- отключение feature flag не возвращает старое поведение;
- появляется запись в склад, ЖПР, оплату или документ без действия в профильном модуле.

## Приемка

Считать готовым можно только когда:

- экран совпадает с утвержденной песочницей;
- read-only агрегатор возвращает контракт v1;
- все спорные связи видны как `needs_review`;
- карта открывает источники, но не проводит учет;
- smoke-цепочки пройдены;
- есть быстрый rollback через feature flag.
