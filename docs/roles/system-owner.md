# Инструкция: Владелец платформы, system_owner

## Назначение роли

`system_owner` обслуживает SaaS-платформу: компании, тарифы, демо-заявки, платежи платформы и системный статус. Это не рабочая роль строительного объекта.

`system_owner` стоит выше всех клиентских аккаунтов. Он видит и управляет структурой:

`platform_account -> companies/legal_entities -> рабочие данные ERP`

Пример: один клиентский аккаунт `Земля Групп` может содержать `ООО Земля 1` и `ООО Земля 2`. Владелец платформы подключает этот аккаунт, выбирает тариф, выдает доступ первому директору и позже может добавить вторую компанию в тот же аккаунт.

## Первый вход

1. Открыть кабинет владельца платформы.
2. Проверить список компаний.
3. Проверить тарифы и демо-заявки.
4. Проверить `Статус системы`.

## Ежедневная работа

- Создавать клиентский аккаунт платформы.
- Создавать первую компанию/юрлицо внутри клиентского аккаунта.
- Подключать дополнительные компании к существующему аккаунту.
- Выдавать инвайт владельцу или директору клиентского аккаунта.
- Менять тариф, срок доступа, демо-статус и статус оплаты.
- Видеть лимиты тарифа: компании, объекты, пользователи, OCR/AI, хранилище.
- Видеть расчетный статус доступа: активен, демо скоро закончится, демо истекло, оплата скоро закончится, просрочка, мягкая заморозка.
- Мягко замораживать неоплаченный аккаунт без удаления компании и без влияния на системную компанию владельца платформы.
- Проверять `Журнал` кабинета владельца платформы: кто создал аккаунт/компанию, менял тариф, продлевал демо, ставил просрочку, замораживал доступ или зачислял оплату.
- Фильтровать журнал по клиентскому аккаунту, компании, действию или тексту, чтобы быстро разбирать спорные действия.
- Смотреть системные ошибки.
- Помогать поддержке, если у клиента проблема с доступом, только через журналируемый режим.

## Что должен видеть результатом

- Компания создана.
- Инвайт выдан директору компании.
- Тариф и срок действия видны в кабинете.
- Демо-срок, просрочка и мягкая заморозка видны отдельными статусами.
- Ключевые действия владельца платформы видны в отдельном журнале платформы, не смешанном с рабочим `audit_log` клиента.
- Если у клиента несколько компаний, они видны внутри одного клиентского аккаунта.
- Клиент может переключаться между своими компаниями, но не видит чужие аккаунты.
- Системные ошибки можно посмотреть без доступа к секретам.

## Нельзя делать

- Не вести ЖПР клиента.
- Не списывать материалы.
- Не менять сметы и финансы клиента без задачи поддержки.
- Не хранить пароли клиентов в открытом виде.
- Не смешивать данные разных клиентов или компаний без фильтра `company_id`.

## Быстрая проверка

Открыть `system_view`: обычный рабочий объект не должен быть основной точкой входа.
