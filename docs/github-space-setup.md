# GitHub Spaces для СтройКа

Эта памятка нужна, чтобы GitHub Copilot/Spaces работал с проектом безопасно и не ломал production-цепочки.

## Зачем это нужно

Space собирает в одном месте контекст проекта: репозиторий, инструкции, важные файлы, правила работы, задачи и вопросы к Copilot. Для нас это полезно как второй слой защиты: Copilot будет видеть правила по деплою, сметам, данным и ролям до того, как предложит правку.

## Как создать Space

1. Открой GitHub Copilot.
2. Слева нажми `Пространства`.
3. Нажми `Создайте пространство`.
4. Название: `СтройКа ERP - production`.
5. Добавь репозиторий: `marikol444707-ux/stroyka-app`.
6. Добавь важные файлы:
   - `.github/copilot-instructions.md`
   - `ONBOARDING.md`
   - `backend/main.py`
   - `src/App.js`
   - `deploy.sh`
   - `scripts/prod-smoke-check.sh`
   - `scripts/check-smeta-parser.py`
7. В описание или инструкции Space вставь блок ниже.

## Текст Для Space

```text
Ты помогаешь с production ERP СтройКа для строительной компании.

Главные правила:
- production деплоится только из ветки main;
- нельзя деплоить claude/*, codex/*, временные ветки и detached HEAD;
- перед деплоем проверить git branch --show-current, git rev-parse --short HEAD, git log -1 --oneline;
- после деплоя выполнить smoke-check;
- нельзя удалять, архивировать или массово менять production-данные без явной команды директора и свежего бэкапа;
- проекты не архивируются автоматически, только вручную директором;
- не трогать .env, пароли, токены и секреты;
- при правках импорта смет обязательно учитывать исходные единицы Excel, коэффициенты количества и итоговые суммы строк;
- 100 м2 x 72,1 должно попадать в систему как 7210 м2;
- нельзя применять один общий коэффициент ко всей смете, потому что мастера и субподрядчики закрывают работы по строкам и объемам;
- материалы и нормы должны быть связаны со снабжением, складом, списанием и бухгалтерией;
- при изменениях парсера смет запускать npm run check:smeta;
- при изменениях фронта запускать npm run build;
- при изменениях backend запускать python3 -m py_compile backend/main.py.

Если предлагаешь изменение, сначала объясни на какие цепочки оно влияет:
смета -> работы -> ЖПР -> КС-2/КС-3 -> бухгалтерия;
смета/нормы -> материалы -> снабжение -> накладная -> склад -> списание -> бухгалтерия;
договор -> акт -> оплата -> удержание.
```

## Какие Spaces Сделать Потом

Сначала достаточно одного основного Space: `СтройКа ERP - production`.

Позже можно добавить отдельные пространства:

- `СтройКа - сметы и материалы`: только импорт смет, нормы, материалы, снабжение, склад.
- `СтройКа - договоры и документы`: шаблоны договоров, акты, КС-2, КС-3, АОСР.
- `СтройКа - мобильная версия`: быстрые экраны для мастера, прораба, снабжения.

## Что Можно Спрашивать В Space

```text
Проверь последний коммит и скажи, какие цепочки он может задеть.
```

```text
Сравни текущую ветку с main и найди рискованные изменения.
```

```text
Проверь, не сломает ли эта правка импорт смет и материалы.
```

```text
Составь короткий issue для исправления этой ошибки.
```

```text
Проверь, какие проверки нужно выполнить перед деплоем.
```

## Чего Не Делать В Space

- Не вставлять пароли, токены, `.env`, ключи S3/Yandex/OpenAI.
- Не просить Space напрямую менять production-базу.
- Не деплоить предложения из Space без проверки ветки и сборки.
- Не принимать правки, которые удаляют данные или архивируют объекты автоматически.

## Минимальный Чеклист Деплоя

На сервере перед деплоем:

```bash
cd /var/www/stroyka-app
git branch --show-current
git rev-parse --short HEAD
git log -1 --oneline
git pull --ff-only origin main
PYTHONPYCACHEPREFIX=/tmp/stroyka-pycache python3 -m py_compile backend/main.py
npm run build
bash deploy.sh
bash scripts/prod-smoke-check.sh
```

Если менялся импорт смет:

```bash
npm run check:smeta
```
