## Суть проекта

`stroyka-app` - система управления строительной компанией.

Главная цепочка:

`объект -> обмеры/проект -> сметы -> пакеты работ -> исполнители -> ЖПР/журналы -> АОСР/КС -> материалы -> финансы -> бухгалтерия`

Смета заказчика должна быть источником правды для работ, материалов, КС и экономики. Для объектов без ГРАНД-сметы допускается отдельный режим работы по внутреннему прайсу.

## Целевая цепочка исполнения

### 1. Объект и сметы

- У объекта может быть несколько сметных пакетов: `общестрой`, `отделка`, `электрика`, `слаботочка`, `отопление`, `вентиляция`, `ВК/канализация`, `благоустройство` и т.д.
- Базовая единица исполнения - не вся смета и не старый наряд, а `пакет работ` внутри объекта.
- Заказчик через API и кабинет видит только активные сметы типа `Заказчик` по своему объекту. Внутренние, материальные, исполнительские и черновые сметы заказчику не отдавать.
- Материальные/ресурсные строки сметы с количеством `< 0` или суммой `< 0` считать корректировкой/исключением: учитывать в общей сумме сметы, но не включать в графу `Материалы`, заявки, потребность, контроль остатков и списание. Строки с количеством `0` не формируют заявки/списание, пока нет реального количества.
- Покрытие нормы материала считать только по положительным ресурсным строкам. Если по норме нужно больше, чем есть положительного материала в смете, статус должен быть `Нехватка материала по норме`; отрицательные строки не закрывают норму.
- При списании через ЖПР норму материала должен подтверждать сервер. Нельзя доверять `normQuantity`, который пришел только с фронта: если сервер не нашел правило нормы, списание по такой "норме" не принимать.
- Исполнитель получает доступ к связке `объект + пакет`, а не ко всему объекту.
- Мастер, бригада или субподрядчик не должны видеть чужие сметы, чужие пакеты и чужие расценки.
- Цена заказчика и цена исполнения должны храниться отдельно:
  - цена заказчика - для КС, финансов и маржи;
  - цена исполнения - для мастера, субподрядчика, бухгалтера и выплат.

### 2. Помещения и зоны

- Если объект ведется по помещениям, факт работы закрывается по связке `объект + пакет + помещение + строка сметы + исполнитель`.
- Помещение выбирается из справочника объекта, не вводится свободным текстом.
- Для фасада, кровли, наружных сетей и благоустройства вместо помещения используется зона/участок/фронт работ.
- Окна, двери и откосы должны участвовать в обмерах: их нельзя просто выкинуть из расчета, потому что они могут быть отдельными работами в смете.
- Система должна не давать закрыть в помещении больше объема, чем допускают обмеры и уже подтвержденное выполнение.

### 3. Факт выполнения

- Исполнитель вводит объем один раз.
- Из этого факта система автоматически ведет:
  - ЖПР;
  - профильные журналы;
  - историю помещения/зоны;
  - кандидатов в АОСР;
  - накопление выполненных объемов;
  - основу для актов исполнителя;
  - управленческую аналитику.
- Одна и та же работа в одном помещении не должна закрываться двумя исполнителями без явного разбиения по объему, этапу, зоне или операции.

### 4. Материалы

Материал должен проходить полный путь:

`смета/норма -> потребность -> заявка -> приход -> перемещение на объект -> выдача исполнителю -> подпись получения -> списание через ЖПР`

Правила:

- Нормы расхода не заменяют физическую выдачу материала.
- Исполнитель не должен списывать материал, который ему не выдали.
- Выдача материала исполнителю должна быть привязана к объекту, пакету и исполнителю.
- Выдача материала разрешена только конкретному активному пользователю с ролью `мастер`, `бригадир` или `субподрядчик`. Абстрактной `бригаде` материал не выдавать: если материал получает бригада, в системе должен быть конкретный бригадир/субподрядчик с доступом к объекту и пакету.
- Перед выдачей материала исполнителю сервер обязан повторно проверить сметный контроль по объекту и пакету. Передача со склада объекта в личный остаток исполнителя не считается новой закупкой: общий покрытый объем не меняется, но `вне сметы` и `сверх лимита` должны блокироваться.
- Основной контролируемый приход материала на объект идет через цепочку `заявка -> КП -> отгрузка -> приемка`: приемка автоматически создает накладную, историю снабжения, входной контроль и остаток материала на объекте.
- Ручная приходная накладная на объект разрешена только для прямого заказа/распоряжения директора или материала, пришедшего вне цепочки снабжения. Такая накладная обязана сохраняться как `sourceType=manual_project_invoice`, иметь объект, пакет работ, поставщика, номер/дату и позиции, проходить сметный контроль по объекту/пакету и создавать остаток объекта, складскую историю и входной контроль. Ей нельзя дублировать накладную, которая уже создана автоматической приемкой поставки.
- Если ручная объектная накладная содержит материал, которого нет в активной смете или который превышает остаток по смете, директор/замдиректора/снабженец/кладовщик могут принять его как прямой заказ с контрольной пометкой `manualProjectInvoiceOverride`. Это не должно попадать в `Мои траты`, не должно создавать заявку задним числом и не снимает обязанность сметчика/директора решить: доп. смета, расходник, укрупненная работа или ошибка распознавания.
- Прораб принимает ручную или Telegram-накладную только на закрепленный объектный склад. Быстрое действие для прораба обязано автоматически подставлять его объект, основной склад ему закрыт. Если материал не сопоставился со сметой, приход все равно фиксирует факт на объекте с контрольной пометкой и не попадает в `Мои траты`.
- Быстрое действие `Принять на склад` должно открывать фото-скан накладной: пользователь сначала выбирает `Основной склад` или объект, ИИ распознает документ, форма сохраняет выбранный склад/объект и прикрепляет фото накладной в `photos/photoUrl`. Такая операция всегда остается складской накладной и не должна попадать в `Мои траты`.
- Название материала в накладной почти никогда не обязано дословно совпадать со сметой. Серверный контроль обязан сначала применять алиасы, затем безопасное fuzzy-сопоставление по нормализованным словам, единице измерения и активному пакету сметы. Если пользователь оставил пакет `Основная`, а материал найден в другом активном пакете объекта, позиция должна автоматически лечь в найденный пакет. Минусовые/ошибочные строки сметы в потребность не включать.
- Telegram-бот для складской накладной обязан вызывать `/telegram/warehouse-invoices`. Endpoint `/telegram/own-expenses` предназначен только для личных трат и должен отклонять payload с признаками склада/накладной, чтобы выбор `Склад` не записывался в `Мои траты`.
- Приход, перемещение и выдача должны искать складскую строку не только по точному названию, но и по алиасу/безопасному похожему названию с совместимой единицей. Типовые пары вроде `керамогранит`/`гранит керамический`, `ГКЛ`/`лист гипсокартонный`, `ГВЛ`/`гипсоволокно`, крепеж/дюбели/скрепы/кронштейны/направляющие должны стыковаться без ручного переименования.
- Если материал был оприходован на объект как `Основная`, а при выдаче или перемещении сметный контроль нашел реальный пакет работ, сервер должен использовать найденный пакет и искать остаток с безопасным fallback по объекту. Это нужно для старых накладных и распознанных документов, где пакет не был выбран точно.
- В заявках снабжения исполнители (`мастер`, `бригадир`, `субподрядчик`) видят контроль объёмов/лимитов по смете, но не видят сметные деньги заказчика (`plannedSum`, `plannedWorkSum`). Денежный контроль остаётся у директора, замдиректора, снабжения, склада и бухгалтерии.
- В балансе списания материалов учитывать только действующие записи ЖПР. `Отклонено` и `Аннулировано` не должны съедать личный остаток исполнителя и не должны блокировать повторную работу по помещению.
- Материалы нужно сверять со сметой, накладными, перемещениями, выдачей, списанием и остатком.
- Себестоимость материалов объекта считать по фактическому приходу (`накладные/поставки/история снабжения`), а не по текущему остатку `materials`. Остаток нужен для складского контроля; после выдачи/списания он уменьшается, но расход объекта уже состоялся и не должен исчезать из экономики.
- Если материал из накладной не выделен отдельной ресурсной строкой сметы, но относится к укрупненной работе сметы (`вентилируемый фасад`, `кровля`, `инженерная система` и т.п.), его нужно привязать к этой работе как `Комплектацию укрупненной работы`. Такая строка не считается `вне сметы`, не блокирует приход/заявку, но не считается обычным ресурсом сметы и не закрывает отдельную сметную потребность. Она обязана оставаться видимой в фактических затратах, поставщике, объекте, пакете работ и бухгалтерии.
- Если материал не найден в смете и не привязан к укрупненной работе, он считается `вне сметы` и требует проверки/доп. сметы/управленческого решения. Для заявок снабжения это блокировка; для ручной объектной накладной по прямому заказу это предупреждение с `manualProjectInvoiceOverride`.

### 5. Бухгалтерия и договоры

- Бухгалтер должен получать не разрозненные данные, а готовый реестр закрытия по исполнителю, объекту и пакету.
- Один субподрядчик может работать на нескольких объектах: карточка исполнителя общая, договоры/акты/платежи привязаны к объектам.
- Месячное закрытие должно собираться из подтвержденных объемов ЖПР и внутренних расценок.
- Удержание 5 процентов на 1 год должно отражаться в договоре, акте, оплате и остатке.
- `Мои траты`: `own_expenses` хранит реестр возмещения сотруднику, `expenses` хранит фактический финансовый расход. При создании траты из веба или Telegram backend автоматически создает связанную строку в `expenses`; при статусе `Отклонено` связанный расход удаляется из экономики объекта; при возврате в рабочий/возмещенный статус расход создается снова. Telegram-доступ идет только через `X-Telegram-Bot-Token` и зарегистрированный Telegram ID/chat ID сотрудника.
- Бухгалтерская сводка: факт оплаченных денег считать из `project_payments` плюс подотчетные, а счета поставщиков и акты исполнителей использовать как основание/долг. Нельзя повторно прибавлять `supplier_invoices.paid_amount` или `interim_acts.paid_amount`, если оплата уже отражена в платежном журнале.

### 6. График руководителя

График объекта должен строиться автоматически из:

- активных смет и пакетов;
- обмеров помещений/зон;
- назначенных исполнителей;
- факта выполнения по ЖПР;
- АОСР и этапов контроля;
- прихода, перемещения и выдачи материалов;
- дедлайнов объекта и этапов.

Руководитель должен видеть план, факт, отставания, блокировки по материалам, проблемные помещения и зависимые этапы.

## Жесткие правила

- Меню роли и backend API должны совпадать. Если раздел скрыт в меню, прямой API-запрос тоже должен быть закрыт. Прайс-листы доступны только ролям, которым они нужны по работе; исполнители видят прайс только на объектах без активной сметы заказчика.
- Не удалять и не архивировать объект обычной кнопкой, `DELETE /projects` или обычным `PUT /projects`.
- Объект закрывается только отдельной процедурой директора: бэкап, проверка документов, проверка смет/финансов, затем явное закрытие.
- Не переводить старую активную смету в архив автоматически при загрузке или активации новой версии. Старая версия должна оставаться видимой как черновик/история группы.
- Безвозвратное удаление сметы разрешено только директору и только если смета не связана с ЖПР, АОСР, договорами, материалами, заявками, поставками и накладными. Для остальных ролей использовать `Снять активность` или `Черновик`.
- Сотрудников не удалять физически из production. Отключение сотрудника = статус `Уволен` + выключенный пользователь, чтобы договоры, акты и история не теряли ссылку на исполнителя.
- `scripts/prelaunch-cleanup.sh` не должен чистить production. Разрешен только `DRY_RUN=1`.
- Тестовые артефакты `CODEX QA` убирать только через `scripts/manage-test-artifacts.py`: сначала dry-run, затем явное `--apply --confirm CODEX_QA`. Скрипт не удаляет данные, а переводит тестовые сметы в `Черновик`, отменяет тестовые заявки, аннулирует/отклоняет тестовые ЖПР/акты/договоры/накладные; тестовых пользователей отключает только при отдельном флаге `--disable-test-users`.
- Перед любым действием с боевой БД создать бэкап и понимать, как откатиться.
- Не делать общий коэффициент на всю смету. Мастерам платят по конкретной строке и фактическому объему.
- Старые ручные наряды не развивать. Новые работы идут от сметы или прайсового объекта.
- Удаление подписанных передач материалов запрещено. Нужен отдельный возврат.
- Если список длиннее 10 элементов, нужен поиск/фильтр.
- Nginx backup класть только в `/etc/nginx/backups/`, не в `/etc/nginx/sites-enabled/`.
- `outputs/`, `tmp/`, локальные тестовые выгрузки и секреты не коммитить.

## Что нужно доделать

### A. Проверка и запуск на живом объекте

- После каждого деплоя запускать `smoke:prod`; после крупных правок - `smoke:core-chains` или `smoke:preprod`.
- После любых действий с боевой БД запускать `smoke:data-guard`, чтобы не потерять объект, сметы, сотрудников, пользователей и директора.
- Перед финальным production-запуском запускать `smoke:test-artifacts`. Если активны только тестовые пользователи - оставить их до конца тестов; перед запуском отключить через `--disable-test-users`.
- Прогнать руками живые входы и ключевые действия по ролям: директор, замы, главный инженер, прораб, мастер, сметчик, снабженец, кладовщик, бухгалтер, технадзор, заказчик.
- Проверить, что меню роли и API-доступы не расходятся после следующих правок.

### B. Сметы и версии

- Не ломать уже сделанные правила импорта: множители единиц (`100 м2`, `1000 шт`), отрицательные ресурсные строки, разделение работ/материалов, суммы из файла.
- Если на реальных файлах снова всплывут спорные кейсы по грунтовке, блокам, втулкам, светильникам, креплениям, радиаторам, отоплению или электрике - править точечно через диагностику парсера.
- Если итог файла не сходится с системой, показывать диагностическую строку, но не отдавать ее мастеру и не создавать из нее материал.
- ССР хранить как документ, но не импортировать в выполнение, если нет строк работ и ресурсов.
- При загрузке новой версии сметы автоматически сравнивать со старой и печатать сопоставительную ведомость.
- Если доп. работа появилась в новой смете, связать ее с ранее внесенной работой вне сметы.
- ✅ В карточке сметы добавлен первый слой `Свод одинаковых работ`: одинаковые работы собираются в одну строку по смете/пакету/названию/единице/типу обмера, с раскрытием исходных строк.
- `Свод работ` не заменяет исходную смету и не переписывает строки импорта. Исходная смета остается юридическим/расчетным источником, а свод нужен для назначения исполнителей, цен, контроля объемов и дальнейших актов.
- Ключ группировки работы: объект, активная смета/пакет работ, нормализованное название работы, единица измерения и тип обмера/основание. Работы с разными единицами или неоднозначным основанием не объединять автоматически; отправлять в `Требует проверки`.
- В агрегированной строке хранить общий объем, сумму заказчика, расчетную цену заказчика за единицу, цену/коэффициент исполнителя, назначенного исполнителя и статус.
- В каждой агрегированной строке обязательно хранить ссылки на исходники: `estimateId`, версия, раздел, индекс/номер строки, исходное название, исходный объем, единица, цена, сумма. Из свода должна быть кнопка `К строкам сметы`.
- Директор или зам назначает цену исполнителю один раз на агрегированную работу; система применяет ее к связанным исходным строкам для договоров, ЖПР, актов и расчетов, но не показывает исполнителю смету заказчика.
- Мастер, бригадир и субподрядчик видят только свой пакет/агрегированную работу/доступный объем/помещения и свою цену/заработок. Сметные суммы заказчика и чужие работы не показывать.
- Если один и тот же текст работы встречается в разных пакетах или с разной логикой обмера, объединять только внутри пакета. Межпакетная агрегация допускается только как отдельный отчет директора, без назначения исполнителя.

### C. Нормы материалов и AI

- Проверить на живом объекте разделение норм: глобальные нормы компании, объектные поправки, сметные поправки.
- Не сохранять слабые AI-предложения в глобальный справочник автоматически.
- Из результата `Вся смета по нормам` формировать черновик доп. материалов или доп. работ с ценами из прайс-листа, если цены найдены.
- Не менять активную смету заказчика автоматически.

### D. Работы, ЖПР, АОСР, КС

- Прогнать на живом объекте цепочку `смета -> сделано -> ЖПР -> АОСР -> КС-2/КС-3 -> финансы`.
- Проверить автопопадание факта выполнения в профильные журналы на реальных ролях.
- AI должен помечать в смете работы, которые потенциально требуют АОСР.
- Работы сверх сметы оформлять как изменения к смете.

### E. Роли и кабинеты

- Проверить кабинет прораба на живом объекте: видит основные сметы своего объекта и выполнение по помещениям/зонам, но не видит чужие объекты.
- Проверить карточку исполнителя на живом сценарии: роль доступа, специализация, тип исполнителя, доступные объекты, доступные пакеты, схема оплаты.
- Регистрация персонала и поставщиков должна идти через заявку/приглашение с подтверждением директора или замдиректора. Саморегистрация с сайта не должна сразу открывать полный доступ к ERP.
- Поставщик после регистрации должен попадать в систему как `на проверке`: карточка поставщика, реквизиты, документы, категории поставок. Активным для КП, поставок и закрытых аукционов он становится только после подтверждения.
- Персонал нужно показывать не одной общей строкой, а группами: руководство, ИТР/управление, прорабы, мастера, бригадиры, субподрядчики, снабжение/склад, бухгалтерия, заказчик/технадзор.
- Внутри мастеров, бригадиров и субподрядчиков обязательна специализация: общестрой, отделка, электрика, слаботочка, отопление, вентиляция, ВК/канализация, благоустройство или другая пакетная категория объекта.
- Исполнитель может иметь доступ к нескольким объектам и нескольким пакетам работ, но не должен видеть чужие сметы, чужие ставки и чужие закрытые работы.
- Закрытые аукционы для мастеров/бригад/субподрядчиков должны опираться на эту структуру: объект -> пакет работ -> объем/срок -> список допущенных исполнителей по категории и специализации -> предложения -> выбор директора/зама.
- Кабинет владельца платформы проверить отдельно: компании, тарифы, пользователи, статус системы, изоляция данных.
- Заказчик и технадзор не обязаны подписывать все в системе, но должны иметь просмотр, печать и замечания.

### F. Договоры и бухгалтерия

- Проверить на живом объекте, что акт, удержание 5%, авансы, оплаты и остаток корректно сходятся после реальной выплаты.
- На проде прогнать бухгалтерскую цепочку на живом объекте: счет поставщика -> оплата -> платежный журнал; акт исполнителя -> оплата -> платежный журнал; свои траты -> возмещение -> платежный журнал.
- Проверить, что факт расходов не задваивается: `project_payments` - факт оплаты, счета поставщиков и акты исполнителей - основание/долг.
- Проверить печатные формы договоров, актов, реестров и удержания перед реальной выдачей подрядчикам.

### G. Продакшен-готовность

- Ближайший порядок работы перед тестовым запуском: сначала ускорение больших экранов и мобильной PWA, затем доработка `Свод работ по смете`, затем проверка живых ролей и цепочек. Новую бизнес-логику добавлять только после того, как основные экраны открываются быстро и без зависаний.
- Тестовый запуск на живых людях разрешать поэтапно: сначала директор/замы/прораб на чтение и проверку данных, затем снабжение/склад, затем мастера/субподрядчики для ЖПР и объемов, затем бухгалтерия и заказчик/технадзор.
- На период тестового запуска не удалять и не архивировать реальные объекты, сметы, пользователей, накладные и журналы автоматически. Любое закрытие/архивация - только явным действием директора.
- Sentry или аналог для ошибок frontend/backend.
- 2FA для директора, замов и бухгалтера.
- Миграция старых локальных `/uploads` в S3.
- PDF-экспорт через серверную библиотеку вместо HTML-печати.
- PWA push или Telegram для поручений, предписаний, проверок, поставок и перерасхода.
- Мобильная PWA должна оставаться в мобильном режиме на телефоне даже в горизонтальной ориентации: склад, материалы, накладные, уведомления и быстрые действия показывать карточками/адаптивными сетками, без горизонтального уезда колонок за экран.
- Полноценное мобильное приложение делать после стабилизации PWA и API.
- Продолжить рефакторинг `src/App.js` и `backend/main.py`, не смешивая рефакторинг с изменением бизнес-логики.
- Производительность больших смет: не рендерить полную смету целиком по умолчанию. Первый экран должен показывать объект, пакеты, свод работ и свод материалов; исходные строки сметы открывать по требованию, с пагинацией/виртуальным списком.
- Тяжелые расчеты по сметам, материалам, своду работ, суммам и нормам выполнять на backend и кешировать результат по `estimateId/version/project/workPackage`, чтобы React не пересчитывал тысячи строк на каждом открытии страницы.
- Для больших таблиц использовать серверные `limit/offset` или курсор, поиск на backend, индексы в БД и отдельные lightweight endpoints для мобильной версии.
- Тяжелые разделы интерфейса грузить лениво: сметы, склад, бухгалтерия, CRM, AI-контроль, печатные формы. На телефоне показывать карточки и короткие списки вместо широких таблиц.
- Если OCR/AI/импорт смет работают на том же сервере, production-сервер должен быть не ниже `2 vCPU / 4 GB RAM`; комфортнее `4 vCPU / 8 GB RAM` и отдельный worker для OCR, AI, импорта и ночных проверок.
- Yandex Workflows пока оставить как эксперимент, не встраивать в основную производственную цепочку. Возвращаться к нему только под конкретную задачу: утренний отчет директору, внешний мониторинг, пакетная обработка документов или отдельная интеграция. Склад, накладные, материалы и траты должны продолжать работать внутри `stroyka-app` без зависимости от Workflow.

### H. Публичная регистрация и закрытые аукционы

Что Codex может сделать сам:

- Добавить публичные формы заявок с сайта: поставщик, мастер, бригадир, субподрядчик, прораб, главный инженер и другие роли.
- Сделать таблицу/статусы заявок регистрации: новая, на проверке, подтверждена, отклонена, заблокирована.
- Переделать текущую регистрацию поставщика: вместо мгновенного `Активный` ставить `На проверке`, пока директор или замдиректора не подтвердит.
- Разбить вкладку `Персонал` на группы и подгруппы по роли, типу исполнителя, специализации, объектам и пакетам работ.
- Добавить фильтры персонала: объект, пакет работ, специализация, статус доступа, тип оплаты, документы.
- Связать регистрацию с доступом: после подтверждения автоматически создавать/обновлять пользователя, карточку `staff` или `suppliers`, назначать объекты и пакеты.
- Заложить основу закрытых аукционов: объект, пакет работ, объем, срок, допущенные исполнители, ставки, история выбора, связь с договором и актами.
- Написать smoke-проверки: поставщик регистрируется и не видит ERP до подтверждения; мастер видит только свой пакет; директор подтверждает заявку; аукцион не показывает лишние сметы.

Что должен решить владелец продукта:

- Самостоятельно с сайта могут подавать заявки: поставщики, субподрядчики, мастера, бригадиры.
- Поставщик заполняет расширенные реквизиты по уже существующей логике поставщика: компания, контакт, телефон, email, сайт, ИНН, КПП, ОГРН, адрес, банк, БИК, счет, категория поставок/материалов.
- Мастер, бригадир и субподрядчик заполняют профиль исполнителя: ФИО/название, телефон, email, сайт/соцсети при наличии, тип исполнителя, специализация, какие работы выполняет, опыт, примечание.
- Документы и подписанный договор можно загружать после подтверждения заявки в карточке исполнителя/поставщика.
- Подтверждать заявки могут директор и заместители директора.
- Категории работ для исполнителей - это не отдельная жесткая смета, а специализации/пакеты работ: общестрой, отделка, электрика, слаботочка, отопление, вентиляция, ВК/канализация, благоустройство и свободное поле `другое`, если нужной категории нет.
- Закрытые аукционы пока не делать как полноценный продукт. Заложить структуру данных под будущее, но внедрять позже, когда будет база исполнителей примерно 100+ компаний/мастеров/бригад.
- Будущая логика аукционов: директор/зам отправляет работу подходящим исполнителям, исполнитель видит только работу, объем, срок и свою цену; смету заказчика и чужие ставки не показывать. После принятия работы система готовит договор и связывает его с дальнейшими актами.

## Публичный сайт и калькулятор

Актуальная цель: довести публичный сайт `stroyka26.pro` до рекламного запуска, но не путать его с рабочей ERP.

Рекомендуемая доменная схема:

- `stroyka26.pro` - публичный сайт, калькулятор, галерея, заявки;
- `app.stroyka26.pro` - рабочая ERP для сотрудников, мастеров, бухгалтерии и клиентов;
- `api.stroyka26.pro` - опционально позже, если понадобится отдельный API-домен.

Если покупаются дополнительные домены, не делать на каждом отдельный сайт. Правильная схема:

- один основной публичный домен для рекламы и SEO: `stroyka26.pro`;
- все похожие домены (`.ru`, `.рф`, короткие названия, домены под рекламу) ставить 301-редиректом на основной публичный домен;
- рабочую ERP не размещать на рекламном домене, а держать отдельно на `app.stroyka26.pro`;
- API не открывать как отдельный публичный сайт; `api.stroyka26.pro` нужен только если понадобится техническое разделение;
- не плодить дубли страниц на разных доменах, иначе SEO будет размазываться и поисковики будут видеть копии.

Что уже есть:

- публичный сайт в `src/components/PublicSitePage.jsx`;
