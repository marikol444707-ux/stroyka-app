# ONBOARDING - stroyka-app

Обновлено: 2026-06-13.

Этот файл хранит только то, что еще нужно доработать или помнить для безопасной разработки. Историю выполненных работ, списки коммитов и закрытые этапы сюда не возвращать. Когда пункт реализован, проверен и задеплоен, его нужно удалить из этого файла в том же коммите.

## Контекст

`stroyka-app` - система управления строительной компанией. Главная цепочка продукта:

`объект -> проект/обмеры -> смета -> пакеты работ -> работы -> материалы -> ЖПР/АОСР -> КС-2/КС-3 -> финансы`

Смета должна быть источником правды. Нельзя строить параллельные ручные процессы, которые дублируют смету, материалы или деньги.

## Целевая модель исполнения

Это рабочая схема, от которой дальше строится разработка. Новые функции не должны ей противоречить.

Текущий блок в работе: довести пакетную модель до денег и отчетов. База уже есть: `assignedPackages` хранится у пользователей, карточка сотрудника назначает пакеты, кабинет мастера/субподрядчика показывает работы по доступному пакету, ЖПР сохраняет пакет, помещение, поверхность, ключ строки сметы и отдельные поля цены заказчика/цены исполнения. Следующие незакрытые места: UI настройки внутренней цены директором по пакету/строке, деплой и проверка сохранения месячных актов исполнителей, выплаты исполнителям, отчеты руководителя и автоматический график.

- У объекта есть два режима:
  - `сметный объект` - основа работы идет от загруженных смет заказчика;
  - `объект по прайсу` - для объектов без ГРАНД-сметы, где работа идет только по внутренним расценкам.
- Один объект может иметь несколько сметных пакетов: `общестрой`, `отделка`, `электрика`, `слаботочка`, `отопление`, `вентиляция`, `ВК/канализация`, `благоустройство` и т.д.
- Базовая единица исполнения не вся смета и не старый наряд, а `пакет работ` внутри объекта.
- Для внутренних работ базовая привязка исполнения должна быть не только к пакету, но и к помещению/зоне, если по объекту ведутся обмеры помещений.
- Доступ сотруднику выдается не просто к объекту, а к связке `объект + пакет(ы)`. Для мастеров, бригад и субподрядчиков разрешено назначение сразу на несколько объектов и несколько пакетов.
- Мастер, бригада или субподрядчик не должны видеть все сметы объекта. Они видят только:
  - свои пакеты;
  - свои строки работ;
  - свои внутренние расценки;
  - свой дневной и накопленный заработок.
- Цена заказчика и цена исполнения - это разные сущности:
  - `сметная цена заказчика` нужна для маржи, КС, финансов и сверки;
  - `внутренняя цена исполнения` нужна для мастера, бригады, субподрядчика и бухгалтерии.
- Внутренняя цена исполнения задается директором или уполномоченным руководителем по пакету:
  - коэффициентом от сметы;
  - своей фиксированной расценкой;
  - смешанным режимом по строкам.
- После ввода объема исполнитель сразу видит, сколько заработал за день по своим строкам. Директор видит валовую выработку, маржу и экономию. Бухгалтер видит основу для месячного акта/начисления.
- Если по объекту включены помещения и обмеры, мастер должен указывать, в каком помещении выполнена работа. Без этого строка не должна полноценно закрываться в выполнение.
- Один факт выполнения должен создаваться один раз и дальше использоваться во всей цепочке документов и журналов, чтобы не было ручного задвоения.
- Для работ по помещениям нужен контроль эксклюзивности: одна и та же строка работы в одном и том же помещении не должна бесконтрольно закрываться несколькими исполнителями.
- Автоматизация должна быть приоритетом. Если систему можно завести от событий (`загрузка сметы`, `назначение пакета`, `ввод объема`, `приход материала`, `подтверждение работы`), лишние кнопки не добавлять.

### Ключевые договоренности по цепочке исполнения

Это не идеи, а зафиксированная целевая логика системы. Дальнейшие доработки должны ей соответствовать.

1. Источник правды

- Смета заказчика - основа плановых работ, материалов, КС и управленческой экономики.
- Для объектов без сметы заказчика разрешен отдельный режим работы по внутреннему прайсу.
- Старые ручные наряды не возвращать в архитектуру. Исполнение должно идти от сметы или от прайсового объекта.

2. Пакетная модель

- Один объект может содержать несколько пакетов: `общестрой`, `отделка`, `электрика`, `слаботочка`, `отопление`, `вентиляция`, `ВК/канализация`, `благоустройство` и т.д.
- Базовая единица управления исполнением - `пакет работ`.
- Доступ исполнителю дается по связке `объект + пакет`, а не просто на весь объект.

3. Роли исполнителей

- В `Персонале` люди должны быть разделены по ролям и по производственным специализациям.
- Один мастер, бригада или субподрядчик может быть назначен на несколько объектов и на несколько пакетов.
- Исполнитель не должен видеть чужие пакеты, чужие сметы и чужие расценки.
- По сметным объектам:
  - вентиляция видит только вентиляцию;
  - электрик видит только электрику;
  - сантехник видит только отопление/ВК;
  - отделочник видит только свои отделочные работы.

4. Помещения и зоны

- Если объект ведется по помещениям, работа закрывается только по связке:
  - `объект`;
  - `пакет`;
  - `помещение`;
  - `строка работы`;
  - `исполнитель`.
- Помещение выбирается из справочника помещений объекта, не вводится вручную.
- Если работа относится не к помещению, а к наружному или общему фронту, используется `зона/участок/фронт работ`.
- Помещения и зоны должны быть связаны с обмерами, чтобы нельзя было закрыть лишний объем.

5. Контроль дублей

- Если в помещении работа по строке уже закрыта или закреплена за исполнителем, второй исполнитель не может закрыть ее повторно.
- Без явного разбиения строка считается занятой одним исполнителем.
- Разрешенное совместное выполнение допускается только при явном делении:
  - по количеству;
  - по этапу;
  - по зоне;
  - по отдельной операции.

6. Единый факт выполнения

- Мастер вводит объем один раз.
- Система создает единый `факт выполнения`.
- Этот факт дальше автоматически используется в:
  - ЖПР;
  - профильных журналах;
  - истории помещения/зоны;
  - кандидатах на АОСР;
  - накоплении выполненных объемов;
  - актах исполнителя;
  - управленческой аналитике.

7. Журналы

- Не должно быть повторного ручного ввода одной и той же работы в несколько журналов.
- Маршрутизация должна идти автоматически:
  - общестрой/отделка -> ЖПР;
  - скрытые работы -> ЖПР + кандидат в АОСР;
  - кабельные работы -> ЖПР + кабельный журнал;
  - приемка материалов -> журнал входного контроля;
  - остальные спецработы -> профильный журнал по типу пакета.

8. Деньги исполнителей

- Для каждой строки есть:
  - цена заказчика;
  - внутренняя цена исполнения.
- Внутренняя цена задается директором или уполномоченным руководителем:
  - коэффициентом от сметной цены;
  - фиксированной ценой;
  - смешанным режимом.
- После ввода и подтверждения объема исполнитель должен сразу видеть заработок за день.
- Система должна копить:
  - заработок за день;
  - накопление за месяц;
  - сумму к акту/начислению.

9. Контур директора и бухгалтера

- Директор видит:
  - цену заказчика;
  - внутреннюю цену исполнения;
  - валовую выработку;
  - маржу/экономию;
  - проблемные отклонения.
- Бухгалтер видит:
  - подтвержденные объемы по исполнителю;
  - акты по объектам;
  - удержания;
  - авансы;
  - выплаты;
  - остаток к оплате.
- Основной бухгалтерский реестр закрытия должен строиться по связке:
  - `объект`;
  - `пакет`;
  - `исполнитель`;
  - `помещение/зона` (если применяется);
  - `строка работы`;
  - `подтвержденный объем`;
  - `внутренняя цена исполнения`;
  - `сумма к начислению`;
  - `аванс`;
  - `удержание`;
  - `итого к выплате`.
- Бухгалтер не должен вручную собирать акт из разных мест. Для него система должна автоматически формировать:
  - реестр месячного закрытия по исполнителю из подтвержденного ЖПР;
  - акт выполненных работ исполнителя;
  - ведомость выплат;
  - остаток удержания и заморозки;
  - историю оплат по нескольким объектам одного исполнителя.
- Если один субподрядчик работает на нескольких объектах, бухгалтер должен видеть:
  - отдельное закрытие по каждому объекту;
  - отдельное закрытие по каждому пакету;
  - сводную карточку контрагента по всем объектам.
- Месячное закрытие с исполнителем должно строиться автоматически из подтвержденных объемов, а не вручную с нуля.

10. График руководителя

- Вкладка `График` должна быть не ручной картинкой, а автоматическим производственным графиком объекта.
- График должен собираться из поступающих данных:
  - активных смет и пакетов;
  - обмеров по помещениям/зонам;
  - назначенных исполнителей;
  - факта выполнения по ЖПР;
  - АОСР и этапов контроля;
  - прихода и выдачи материалов;
  - дедлайнов по объекту и этапам.
- Для руководителя график должен показывать:
  - план по пакетам;
  - факт по пакетам;
  - отставание/опережение;
  - проблемные помещения/зоны;
  - зависимость этапов от материалов и смежников.
- Проверить текущую первую сводку вкладки `График`: сроки, этапы, активность ЖПР, прогресс по смете и сигналы внимания должны совпадать с реальным объектом.
- Логика построения графика:
  - если у пакета есть смета и объем, система строит плановый фронт;
  - если есть фактические закрытия по помещениям/зонам, система строит фактический прогресс;
  - если фактический прогресс ниже ожидаемого на дату, создается сигнал руководителю;
  - если материалы не пришли или не выданы, график помечает блокировку пакета;
  - если работа подтверждена, график сдвигает следующий зависимый этап автоматически.
- Ручное редактирование графика допускается только как управленческая корректировка, а не как основной источник факта.

11. Материалы

- Материалы должны считаться и сверяться по пакетам объекта.
- Материал проходит путь:
  - смета/норма;
  - потребность;
  - заявка;
  - приход;
  - перемещение на объект;
  - выдача исполнителю;
  - списание по выполненным работам.
- Исполнитель не должен списывать материал, который ему не выдали.
- Если материал выдан, система должна помнить, под какой объект, пакет и исполнителя он числится.

12. Автоматизация вместо лишних кнопок

- Если данные уже есть в смете, пользователь не должен руками выбирать:
  - единицу;
  - тип работы;
  - основание;
  - режим объема.
- Если событие уже произошло в системе, связанная запись должна появляться автоматически, а не через вторую кнопку.

Основные файлы:

- `backend/main.py` - FastAPI backend, миграции, API, бизнес-логика.
- `src/App.js` - основной frontend-контейнер, постепенно выносится в компоненты.
- `src/components/` - вынесенные UI-компоненты.
- `src/api.js` - общий API helper.
- `scripts/prod-smoke-check.sh` - smoke-проверка прода.
- `scripts/prelaunch-cleanup.sh` и `scripts/prelaunch-cleanup.sql` - только безопасный dry-run.
- `ROLES.md` и `docs/roles/` - инструкции по ролям.

## Жесткие правила

- Не удалять и не архивировать объект обычной кнопкой, обычным `DELETE /projects` или обычным `PUT /projects`.
- Объект закрывается только отдельной процедурой директора: бэкап, проверка документов, проверка смет/финансов, затем явное закрытие.
- `scripts/prelaunch-cleanup.sh` не должен чистить production. Разрешен только `DRY_RUN=1`.
- Перед любым действием с боевой БД: создать бэкап и проверить, что восстановление возможно.
- Не делать общий коэффициент на всю смету. Мастерам платят по конкретной строке и фактическому объему.
- Нормы расхода не заменяют физическую выдачу материалов. Материал должен пройти путь: приход/перемещение на объект -> выдача мастеру -> подпись получения -> списание через ЖПР.
- Удаление подписанных передач материалов запрещено. Нужен отдельный возврат.
- Старые ручные наряды не развивать. Новые работы идут от сметы и изменений к смете.
- Если список длиннее 10 элементов, сразу нужен поиск/фильтр.
- Nginx backup класть только в `/etc/nginx/backups/`, не в `/etc/nginx/sites-enabled/`.
- `outputs/`, `tmp/`, локальные тестовые выгрузки и секреты не коммитить.

## Ближайшая проверка прода

1. Убедиться, что на сервере задеплоен последний код с защитой от случайного удаления объектов.
2. Проверить восстановленные данные:
   - объект `Кисловодск Лицей 4` есть;
   - активные сметы объекта видны;
   - сотрудники и пользователи не потеряны;
   - черновик материалов по нормам не мешает активным сметам.
3. Проверить вход сотрудников, созданных из карточки персонала:
   - пароль меняется без показа старого пароля;
   - существующий email обновляет доступ, а не создает дубль;
   - роль и назначенные объекты попадают в кабинет.
4. Проверить, что прораб видит основные сметы по своему объекту.
5. Проверить `npm run smoke:prod` с логином директора и разобрать `apiErrorsShown`, если число больше нуля.

## Активный план

### 1. Импорт смет как рабочий документ

Цель: файл ГРАНД/Excel после загрузки сразу становится нормальным рабочим документом системы без ручного выбора типа, основания, единиц и странных коэффициентов.

Что довести:

- Повторно прогнать реальные сметы Кисловодска после последних фиксов и сверить итог файла с итогом системы.
- Для каждой импортной строки показывать диагностическую расшифровку: исходная единица, исходный объем, множитель единицы (`100 м2`, `1000 шт`), коэффициент объема, итоговый рабочий объем, исходная сумма, источник суммы.
- Рабочая единица берется из файла. Примеры:
  - `100 м2` и количество `72,1` должны стать `7210 м2`;
  - `100 шт` и количество `2,6` должны стать `260 шт`;
  - `1000 шт` и количество `2,6` должны стать `2600 шт`;
  - ресурс с итоговым количеством `0` не должен попадать в потребность закупки.
- Материалы определять только из ресурсных строк, а не из итоговых строк типа `Материалы` или `Строительные работы`.
- Работы не должны попадать в материалы. Материалы не должны попадать в работы.
- Проверить проблемные примеры пользователя:
  - грунтовка должна определяться как материал, если это ресурс, и как работа, если это операция грунтования;
  - блоки, втулки, светильники, крепления, радиаторы не должны получать объемы вида `2600000000 шт`;
  - отопление и электрика не должны давать огромные числа из-за неверно прочитанной колонки.
- Смета ССР хранится как документ, но не импортируется в выполнение, если нет строк работ и ресурсов.

### 2. Суммы сметы и оплата мастеров

Цель: сумма каждой строки должна быть понятной, а не выведенной общим коэффициентом.

Что довести:

- Для ЛСР собирать сумму родительской работы из дочерних строк `ОТ`, `ЭМ`, `НР`, `СП` и текущих денежных колонок.
- Для контрактной сметы брать построчную сумму из файла.
- Если итоговая сумма файла не сходится, показывать отдельную диагностическую строку сверки, но не отдавать ее мастеру и не создавать из нее материал.
- В UI рядом со строкой показывать: сумма строки, источник суммы, исходный объем, рабочий объем.
- Убрать UX, где пользователь вынужден массово выбирать `шт`, `м2`, `ручной объем`, если эти данные есть в смете.
- Для исполнителей считать отдельно:
  - цену заказчика по строке;
  - внутреннюю цену исполнения по строке;
  - дневной заработок = подтвержденный объем за день x внутренняя цена исполнения.
- В карточке объекта проверить управленческий блок экономики:
  - смета заказчика;
  - закрыто заказчику;
  - подтверждено к оплате исполнителям;
  - материалы;
  - прочие затраты;
  - маржа по закрытому и прогноз по плану.
- В карточке объекта проверить блок связей:
  - сметы, обмеры, ЖПР, материалы, журналы, документы, ИИ-контроль и финансы должны открываться из одного места;
  - проблемные статусы должны подсвечиваться без ручного поиска по вкладкам.
- Не показывать мастеру или субподрядчику чужие пакеты и чужие расценки.

### 3. Нормы материалов и AI

Цель: нормы помогают проверить смету и создать черновик материалов, но не портят активную смету заказчика.

Что довести:

- Повысить качество сопоставления норм: не сохранять слабые предложения автоматически.
- Разделить нормы:
  - глобальная база нормы;
  - поправка по объекту;
  - поправка по конкретной смете.
- Добавить понятный статус уверенности: высокая, средняя, слабая. Слабые предложения только в ручную проверку.
- Из результата `Вся смета по нормам` формировать черновик сметы на доп. материалы или доп. работы с ценами из прайс-листа, если цены найдены.
- Не менять активную смету заказчика автоматически.
- Автоконтроль должен сам создавать задачу в `ИИ-контроль` при загрузке/активации/сохранении сметы, без лишних кнопок в строках.

### 4. Материалы, склад, снабжение

Цель: материал должен прослеживаться от сметы до накладной, склада, выдачи мастеру и списания.

Что довести:

- Сопоставить материалы сметы с приходными накладными.
- В карточке объекта показать аккуратный список всех материалов с прокруткой, фильтром и кнопкой заявки прямо из строки.
- Группировать одинаковые материалы в общую потребность, но сохранять расшифровку источников: смета, раздел, работа, исходная строка.
- Потребность считать по пакетам объекта, чтобы электрик не попадал в отделочные материалы, а вентиляция не смешивалась с сантехникой.
- Сверять:
  - по смете;
  - в заявках;
  - в пути;
  - принято по накладным;
  - перемещено на объект;
  - выдано мастерам;
  - списано;
  - остаток;
  - докупить.
- Если материал пришел вне сметы, сверх плана, с другой единицей или завышенной ценой, создавать задачу сметчику или директору.
- Мастер не должен списывать материал, который ему не выдали и который он не подтвердил.
- Бухгалтеру нужна связка: накладная -> объект -> поставщик -> счет -> материал -> основание по смете.
- Если материал выдан мастеру под пакет, система должна видеть:
  - кому выдан;
  - под какой объект;
  - под какой пакет;
  - сколько уже списано по ЖПР;
  - какой остаток еще числится за исполнителем.

### 5. Работы, ЖПР, АОСР, КС

Цель: выполненные работы идут из сметы, а документы формируются без задвоений.

Что довести:

- Проверить цепочку: смета -> сделано -> ЖПР -> АОСР -> КС-2/КС-3 -> финансы.
- Работы должны попадать исполнителю автоматически из назначенного пакета, без ручного разбрасывания старых нарядов.
- Если у объекта заведены помещения, объем должен закрываться по связке:
  - объект;
  - пакет;
  - помещение/зона;
  - строка работы.
- После сохранения объема система должна автоматически фиксировать этот же факт в:
  - журнале производства работ;
  - связанных спецжурналах по пакету;
  - истории выполнения по помещению/зоне;
  - базе для последующего АОСР/КС/актов исполнителя.
- Мастер должен выбирать помещение из списка помещений объекта, а не вводить его вручную в свободный текст.
- Система должна проверять, что мастер не закрывает по помещению больше, чем допускают обмеры, окна/двери, откосы и ранее подтвержденные объемы.
- Если в помещении уже есть подтвержденная работа по той же строке, второй мастер не должен иметь возможности:
  - закрыть тот же объем повторно;
  - создать дубль той же работы;
  - обойти блокировку через другой журнал или соседний экран.
- Должно быть три состояния контроля по строке в помещении:
  - `свободно` - работа еще никем не начата;
  - `в работе у исполнителя` - объем открыт и резервирован за конкретным мастером/бригадой/субподрядчиком;
  - `закрыто` - подтвержденный объем больше недоступен для повторного закрытия.
- Если по одной строке допускается поэтапное выполнение несколькими людьми, деление должно идти только через явное разбиение на части:
  - по количеству;
  - по этапу;
  - по зоне внутри помещения;
  - по отдельной операции.
- Без такого явного разбиения строка в помещении должна считаться занятой одним исполнителем.
- Для работ, которые не относятся к помещению напрямую (`фасад`, `кровля`, `наружные сети`, `благоустройство`), вместо помещения должна использоваться зона/участок/фронт работ.
- Для профильных журналов нужна автоматическая маршрутизация записи:
  - общестрой/отделка -> ЖПР;
  - скрытые работы -> ЖПР + кандидат в АОСР;
  - электрика/слаботочка по кабелю -> ЖПР + журнал кабельной продукции;
  - материалы/приемка -> журнал входного контроля;
  - спецработы при необходимости -> соответствующий спецжурнал без повторного ручного ввода.
- Если по пакету задана внутренняя цена исполнения, после ввода объема система сразу считает:
  - заработок исполнителя за день;
  - накопление за месяц;
  - валовую выработку по объекту;
  - видимую директору экономию/маржу относительно сметы заказчика.
- АОСР остается исполнительной печатной формой, не финансовым стоп-краном.
- AI должен помечать в смете работы, которые потенциально требуют АОСР, но решение остается проверяемым.
- Работы сверх сметы оформлять как изменения к смете, а не смешивать с плановой сметой.
- При загрузке новой версии сметы автоматически сравнивать со старой и печатать сопоставительную ведомость.
- Если доп. работа появилась в новой смете, связать ее с ранее внесенной работой вне сметы.

### 6. Роли и кабинеты

Цель: каждая роль видит только нужное и может выполнить свой участок цепочки.

Что довести:

- Проверить реальные входы для директора, замов, главного инженера, прораба, мастера, сметчика, снабжения, кладовщика, бухгалтера, технадзора, заказчика.
- Для замов оставить управленческий набор: объекты, сметы, снабжение, материалы, финансы в нужном объеме, ИИ-контроль, персонал без лишних опасных действий.
- Прораб должен видеть основные сметы своего объекта и рабочие документы по ним.
- Прораб должен видеть выполнение не только по пакету, но и по помещениям/зонам, чтобы быстро ловить завышение объема и незакрытые участки.
- В `Персонале` группировать людей по производственным направлениям: `общестрой`, `отделка`, `электрика`, `слаботочка`, `отопление`, `вентиляция`, `ВК/канализация`, `благоустройство` и т.д.
- В карточке сотрудника/исполнителя должны отдельно задаваться:
  - роль доступа;
  - производственная специализация;
  - тип исполнителя: сотрудник, мастер, бригада, субподрядчик, самозанятый;
  - доступные объекты;
  - доступные пакеты внутри объектов;
  - схема оплаты: оклад, сдельная, коэффициент, фикс-прайс.
- Руководящая логика по ролям:
  - директор видит все объекты, все пакеты, цены заказчика, внутренние цены, маржу, финансы;
  - зам директора видит назначенные ему объекты и управленческие показатели без критичных системных действий;
  - прораб видит объект, пакеты объекта, исполнителей, ЖПР, выдачи материалов, замечания;
  - мастер/бригада/субподрядчик видит только свои пакеты, свои работы, свои материалы, свой заработок;
  - бухгалтер видит акты, выплаты, удержания, договоры, закрытие месяца по исполнителю и объекту.
- Для мастеров, бригад и субподрядчиков доступ к нескольким объектам обязателен.
- Кабинет владельца платформы проверить отдельно: компании, тарифы, пользователи, статус системы, без доступа к чужим данным без явного режима администратора.
- Заказчик и технадзор не обязаны подписывать все в системе, но должны иметь удобный просмотр/печать/замечания.

### 7. Договоры и бухгалтерия

Цель: договоры, акты и выплаты связаны с исполнителем, объектом и сметой.

Что довести:

- Самозанятые: отдельный шаблон договора/оферты и актов, без логики ИП/ООО.
- Реквизиты контрагента должны автоматически подставляться в договор.
- У бухгалтера нужна папка по каждому исполнителю и внутри разбивка по объектам.
- Один субподрядчик может работать на нескольких объектах: карточка исполнителя общая, договоры/акты/платежи привязаны к объектам.
- Удержание 5 процентов на 1 год должно быть в основном шаблоне субподрядчика/мастера и понятно отражаться в акте/оплате.
- Месячная цепочка должна быть такой:
  - исполнитель в течение месяца вводит объемы;
  - система копит подтвержденную выработку по внутренним расценкам;
  - директор видит сумму к начислению и маржу;
  - бухгалтер в конце месяца формирует акт/начисление по исполнителю и объекту;
  - удержание, авансы, оплаты и остаток автоматически попадают в карточку исполнителя.

### 8. Продакшен-готовность

Что довести:

- Sentry или аналог для ошибок frontend/backend.
- SMTP или другой нормальный канал восстановления пароля.
- 2FA для директора, замов и бухгалтера.
- Миграция старых локальных `/uploads` в S3.
- PDF-экспорт через серверную библиотеку вместо HTML-печати.
- PWA push или Telegram для поручений, предписаний, работ на проверке, поставок и перерасхода.
- Полноценное мобильное приложение делать только после стабилизации PWA и API.
- Продолжить рефакторинг `src/App.js` и `backend/main.py`, но не смешивать рефакторинг с изменением бизнес-логики.

## Как вести этот файл

- Не писать сюда: "сделано", списки коммитов, историю сессий, старые закрытые фазы.
- Писать сюда только: активная проблема, ожидаемое поведение, где проверить, что нельзя сломать.
- Если задача стала стабильной и прошла проверку на проде, удалить ее из ONBOARDING.
- Если нашли опасное правило, которое предотвращает потерю данных, оставить его в разделе "Жесткие правила".
