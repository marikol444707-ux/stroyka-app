## Суть проекта

`stroyka-app` - ERP для строительной компании.

Главная цепочка:

`объект -> обмеры/проект -> сметы -> пакеты работ -> исполнители -> ЖПР/журналы -> АОСР/акты -> материалы -> финансы -> бухгалтерия`

Смета заказчика - источник правды для работ, материалов, КС и экономики. Для объектов без ГРАНД-сметы допускается режим по внутреннему прайсу.

### Сохранность рабочих данных

- Рабочий объект, активные/загруженные сметы, материалы объекта, основной склад, ЖПР, накладные, оплаты и бухгалтерская история не удаляются физически из базы обычными действиями интерфейса.
- Закрытие объекта - отдельная процедура директора с предварительной проверкой: сметы, ЖПР, материалы, складские движения, АОСР, финансы, бухгалтерия и audit log.
- Лишние сметы и ошибочные версии не удаляются: их снимают из активных, переводят в черновик/служебное состояние или связывают через историю версий, чтобы не потерять основание расчетов.
- Материалы не удаляются кнопкой: для исправления используются корректировка, списание, возврат, перемещение или инвентаризация с записью в историю.
- Прямое удаление/аннулирование выполнения (`DELETE /work-journal/{id}`) и сдельного начисления (`DELETE /piecework/{id}`) доступно только директору. Отклонение ЖПР проверяющим остается отдельным рабочим сценарием через статус `Отклонено`, чтобы прораб мог вернуть работу мастеру без права удаления выплатных данных.
- Перед любым рефакторингом, который касается объекта, смет, склада, материалов, ЖПР, бухгалтерии или доступа, обязательны бэкап БД, `npm run build`, backend compile и smoke-проверка `/app`; после деплоя - production smoke и проверка отсутствия белого/черного экрана.

### Запуск объекта и AI-досье

Запуск объекта выносится в отдельный модуль, чтобы не раздувать `App.js` и `backend/main.py`:

- backend: `backend/features/project_launch/`;
- frontend: `src/features/project-launch/`;
- архитектура: `docs/project-launch-architecture.md`.

Модуль работает по правилу: `файл -> документ объекта -> AI-черновик -> проверка человеком -> запись в рабочие сущности`.

ИИ не должен автоматически менять объект, активировать смету, проводить склад, создавать оплату, архивировать или закрывать объект. Он готовит черновик запуска: заказчик, реквизиты, бюджет, сроки, условия договора, юридические риски, задачи контроля и черновик сметы. В рабочие таблицы данные попадают только после ручного подтверждения.

Текущий внедренный слой: backend-маршруты `/project-launch/*`, таблицы черновиков/условий договора/контрагентов и frontend-вкладка `Запуск объекта` внутри карточки объекта. Вкладка показывает готовность объекта, документы/сметы, создает безопасный черновик из выбранного документа и позволяет отклонить черновик; применения черновика в рабочие сущности пока нет.

## Правила системы

### SaaS-уровни и кабинет владельца платформы

Stroyka должна развиваться как SaaS-платформа поверх текущей ERP, без смешивания рабочих данных разных клиентов.

Главная иерархия:

`system_owner -> platform_account -> companies/legal_entities -> projects/estimates/warehouse/accounting/staff`

Расшифровка:

- `system_owner` - владелец платформы. Это создатель/оператор SaaS, а не директор строительного объекта.
- `platform_account` - клиентский аккаунт или группа компаний, например `Земля Групп`.
- `company` - отдельная строительная компания или юрлицо внутри клиентского аккаунта, например `ООО Земля 1` и `ООО Земля 2`.
- `legal_entity` - реквизиты юрлица, если у одной рабочей компании несколько юридических контуров для договоров, счетов и актов.
- Рабочие данные ERP всегда принадлежат конкретной `company_id`: объекты, сметы, склад, накладные, бухгалтерия, сотрудники, документы и журнал действий.
- Сводка может быть общей по `platform_account`, но любое действие должно выполняться в контексте выбранной компании.

Принятая multi-company архитектура (решение 2026-07-09):

- Каждая независимая строительная организация-клиент получает отдельный `platform_account`. Данные разных `platform_account` нельзя объединять ни ролью директора, ни переключателем компаний, ни общим отчетом.
- Холдинг или группа компаний получает один `platform_account` и несколько `companies`. Одна `company` - один рабочий контур юрлица/организации; внутри нее может быть сколько угодно объектов и площадок.
- Обычный внутренний пользователь принадлежит одному `platform_account`, а доступ к одной или нескольким компаниям получает через `user_company_roles`. Платформенные сотрудники и глобальный кабинет поставщика являются отдельными контролируемыми сценариями, а не обходом границы клиента.
- Глобальная роль в `users.role` остается legacy/default-ролью. Для рабочего действия сервер должен использовать эффективную роль членства в выбранной компании; переход на нее выполняется по одному домену, с тестами совместимости.
- Режим `Все компании` существует только внутри одного `platform_account` и предназначен для сводки, сравнения и контроля. Создание, изменение, проведение, удаление или подписание требует конкретную компанию.
- Объект/стройплощадка всегда принадлежит одной `company_id`. Название объекта не является границей доступа; по мере миграции рабочие цепочки должны опираться на `project_id` и проверенную `company_id` объекта.
- Все новые бизнес-записи получают `company_id` из проверенного сервером контекста или из уже проверенного родительского документа. Клиентский `companyId` и HTTP-заголовки не являются доказательством доступа.
- Поставщик может быть глобальным справочником платформы, но договор, рейтинг, условия, видимость КП и история отношений принадлежат связи `company_supplier_links` конкретной компании-клиента.
- Межфирменное перемещение нельзя маскировать обычным складским движением: это отдельная операция `компания-источник -> компания-получатель` с двумя сторонами, документами и audit log.
- Файлы, уведомления, AI/OCR-задания, audit log, экспорты и фоновые задачи должны наследовать тот же tenant-контекст, что и основной документ.
- Поддержка платформы получает доступ к рабочим данным только через ограниченную support-сессию с причиной, сроком и записью в audit log.

Контракт Tenant Context Kernel:

- Frontend хранит только выбранный UI-контекст и для защищенных запросов передает `X-Company-Mode: company` вместе с `X-Company-Id: <id>` либо `X-Company-Mode: all_companies`.
- Backend считает эти заголовки недоверенным вводом: проверяет формат, членство пользователя, `platform_account`, режим действия и совпадение с компанией объекта/документа.
- Запросы старых клиентов без заголовков временно продолжают работать через default membership и legacy `users.company_id`.
- `Task M1` внедряет транспорт и resolver, а первым потребителем делает создание заявки снабжения. Фильтрация чтения, массовый backfill, ограничения БД и перевод всех ролей на effective role входят в последующие M-задачи.

Кабинет владельца платформы должен показывать:

- все клиентские аккаунты;
- компании/юрлица внутри каждого аккаунта;
- тариф, статус оплаты, демо-срок, дату окончания доступа;
- лимиты тарифа: компании, активные объекты, внутренние пользователи, объем AI/OCR, хранилище;
- количество пользователей, объектов, документов, ошибок, страниц OCR и активность за период;
- демо-заявки, приглашения директоров, статусы подключения;
- платежи платформы и долги клиентов;
- системные ошибки и здоровье сервера без раскрытия секретов;
- audit log действий владельца платформы, администраторов и поддержки.

Кабинет владельца платформы не должен быть рабочим кабинетом стройки:

- не вести ЖПР клиента;
- не проводить накладные;
- не менять сметы, оплаты, остатки и документы клиента без режима поддержки;
- не хранить и не показывать пароли клиентов;
- не обходить `company_id` и роли клиента.

Быстрая загрузка карты клиента в кабинете платформы:

- живет только в SaaS-слое: frontend `src/features/platform-admin/`, backend `/system/client-card/recognize` в `backend/features/platform_admin/`;
- принимает фото, PDF, Word, Excel, TXT/CSV/RTF и другие файлы визитки, карточки организации или реквизитов и возвращает черновик полей для формы подключения аккаунта/компании;
- не создает компанию, аккаунт, пользователя или доступ автоматически;
- найденные поля подставляются в форму, но оператор платформы обязан проверить и нажать сохранение вручную;
- файл карты хранить как служебную загрузку платформы, а факт распознавания писать в audit log.

Будущие роли платформы:

- `system_owner` - полный владелец платформы.
- `platform_admin` - администратор платформы: подключает клиентов, демо, тарифы, компании и пользователей в рамках разрешений.
- `platform_support` - поддержка: видит ошибки и помогает с доступом через режим поддержки и audit log.
- `billing_admin` - платежи платформы, счета, долги, заморозка/разморозка.
- `account_owner` - владелец клиентского аккаунта, видит свои компании и общую сводку.
- `account_admin` - администратор клиента, управляет пользователями и доступами внутри своего аккаунта.

Клиентский аккаунт должен работать так:

- владелец платформы создает `platform_account`, например `Земля Групп`;
- внутри него создает первую компанию, например `ООО Земля 1`;
- выдает инвайт директору/владельцу клиентского аккаунта;
- позже можно добавить вторую компанию, например `ООО Земля 2`;
- клиент видит переключатель `Все компании / ООО Земля 1 / ООО Земля 2`;
- в режиме `Все компании` видна только сводка, а создание/изменение документов требует выбрать конкретную компанию;
- сотрудники могут иметь доступ ко всему аккаунту или только к конкретной компании/объектам.
- пользователи клиентских компаний управляются в отдельной вкладке кабинета платформы: владелец/админ платформы может посмотреть аккаунт, компанию, роль, активность и 2FA, отключить/включить доступ или перенести пользователя в другую компанию того же клиентского аккаунта только с обязательной причиной и записью в audit log.
- управление клиентскими пользователями не показывает пароли, TOTP-секреты и reset-токены, не трогает платформенные роли `system_owner/platform_admin/platform_support/billing_admin`, не переносит пользователя в служебную компанию и при переносе очищает старые объектные/пакетные привязки.

Тарифную модель строить гибридно, не только по пользователям:

- `Демо`: 14-30 дней, 1 компания, 1 объект, 5 пользователей, ограниченный OCR/AI.
- `Старт`: одна компания, несколько активных объектов, базовые ERP-модули.
- `Компания`: 1-2 компании, больше объектов и пользователей, бухгалтерия, снабжение, OCR накладных, роли и сводки.
- `Группа`: несколько компаний/юрлиц, общий кабинет группы, переключатель компаний, расширенный аудит и лимиты.
- `Enterprise`: индивидуальные лимиты, домен, API, SLA, интеграции и отдельные правила поддержки.

Безопасный порядок внедрения SaaS-слоя:

1. Документация и схема данных.
2. Read-only кабинет владельца платформы: аккаунты, компании, тариф, статус, лимиты, ошибки.
3. Создание клиентского аккаунта и первой компании из кабинета владельца платформы.
4. Инвайт директору клиента и базовая привязка пользователей к аккаунту/компании.
5. Подключение второй компании к одному аккаунту.
6. Переключатель контекста `Все компании / компания`.
7. Тарифные лимиты и предупреждения.
8. Демо-доступ, истечение демо, заморозка неоплаченных аккаунтов.
9. Платежи, счета, ЮKassa и отчеты владельца платформы.
10. Администраторы платформы и режим поддержки с audit log.

Техническое правило внедрения:

- Не расширять бесконечно общий `App.js` и `backend/main.py`.
- Frontend SaaS-слой выносить в `src/features/platform-admin/`.
- Backend SaaS-слой постепенно выносить в `backend/features/*`: системный кабинет остается в `backend/features/platform_admin/` с роутами `/system/*`, контекст компаний вынесен в `backend/features/company_context/`. Не создавать пакет `backend/platform`, чтобы не конфликтовать со стандартным Python-модулем `platform`.
- Новые backend-модули строить по схеме `routes.py` для HTTP, `service.py` для бизнес-логики и при необходимости `repository.py` для SQL. `backend/main.py` должен только хранить legacy-роуты, общие зависимости, схему совместимости и регистрацию feature-модулей.
- В существующие рабочие разделы добавлять только обязательный контекст `company_id`, проверки доступа и фильтрацию.
- Первый слой делать read-only и без блокировок, чтобы не сломать текущую ERP.

- Ограничение текущего слоя: `users.company_id` хранит одну default/legacy-компанию и не является источником прав для сценария `один директор -> несколько компаний одного platform_account`. Для этого уже используется совместимая таблица `user_company_roles`; `users.company_id` пока не удаляется.
- Контракт `user_company_roles`: `user_id`, `platform_account_id`, `company_id`, `role`, `assigned_projects`, `assigned_packages`, `active`, `is_default`, `created_at`, `updated_at`. Один пользователь может иметь разные рабочие роли в разных компаниях одного клиентского аккаунта.
- Кабинет владельца платформы должен уметь назначать пользователю несколько компаний и ролей внутри одного `platform_account`, отключать отдельное членство, менять default-компанию и писать все изменения в `platform_audit_log`. Старый перенос пользователя между компаниями оставить совместимым режимом до миграции.
- Рабочий frontend должен получить отдельный модуль `src/features/company-context/`: загрузка доступных компаний пользователя, переключатель `Все компании / компания`, хранение выбранной компании в безопасном UI-state и передача `companyId` в изменяющие запросы.
- Режим `Все компании` разрешает только сводку, контроль и аналитику. Создание/изменение объекта, сметы, заявки, КП, накладной, склада, оплаты, акта или документа требует выбранную конкретную `company_id`.
- Backend должен получить общий helper `resolve_company_context(current_user, requested_company_id, action_mode)`: проверить членство пользователя, роль в выбранной компании, активность компании, принадлежность к `platform_account`, запретить рабочие действия в режиме `all_companies`.
- Все новые рабочие записи должны писать `company_id` явно: проекты, сметы, материалы/ресурсные группы, заявки снабжения, получатели КП, КП, счета поставщика, поставки, складские накладные, складские движения, оплаты, ЖПР, акты и документы. Старые строки с `company_id=1` не переписывать массово без dry-run.
- Наследование `company_id` в снабжении: заявка/КП задают компанию; счет поставщика берет ее из КП или связанной складской накладной; поставка берет ее из КП/заявки; складская накладная и `warehouse_history` при приемке берут ее из поставки/счета/объекта. Если источник старый и компании нет, использовать legacy `1`, но не смешивать будущие дубли по номеру накладной без проверки `company_id`.
- Поставщики остаются глобальными участниками платформы. Связь поставщика с конкретной строительной компанией хранить через `company_supplier_links`; расширить ее под `platform_account_id`, локальную категорию, договор, рейтинг, условия оплаты, источник связи и статус.
- Для отправки КП использовать таблицу `supply_request_recipients`: `company_id`, `request_id`, `supplier_id`, `supplier_group_ids`, `target_supplier_id`, `supplier_user_id`, `visible_to_supplier`, `problem_reason`, `status`, `requested_at`, `responded_at`. Это источник правды, кто реально должен увидеть запрос; endpoint `/supply-requests/{id}/recipients` показывает диагностику видимости.
- `selected_suppliers` и `supplier_offers.supplier_id` оставить как совместимость и историю. Новая цепочка должна идти через `supply_request_recipients -> supplier_offers`; старый путь не удалять до успешных smoke-проверок и backfill.
- Перед созданием КП делать preflight: материал прошел сметный/ресурсный контроль, поставщик связан с компанией, дубли поставщика найдены, выбран target supplier, аккаунт поставщика привязан, кабинет поставщика увидит запрос. Если не увидит - блокировать отправку или показывать директору явную причину.
- Кабинет поставщика перевести на backend-источник истины `/supplier-cabinet/requests`: возвращать запросы, КП, счета, накладные и поставки по всем связанным `supplier_id`, сгруппированные по `company_id`/компании-клиенту. Frontend не должен сам угадывать принадлежность важных документов только по названию.
- Сметный контроль перевести с точного названия на ресурсную модель: `material_resource_groups`, алиасы/бренды/синонимы, характеристики, единицы пересчета, плановый объем, заявлено, закуплено, остаток. Пример: `штукатурка гипсовая` как ресурс, а `Knauf`, `Сновит`, `Хоббест` как фактические товары/алиасы.
- Контроль материалов должен быть светофором, а не глухой стеной: `OK`, `аналог по группе`, `требует проверки`, `вне сметы`, `превышение`, `ошибка раздела`. Жестко блокировать только опасные случаи, остальные отправлять в согласование директора/сметчика с видимой причиной.
- Старые накладные, КП, счета, складские движения и оплаты не удалять и не пересоздавать. Backfill должен только добавлять связи `company_id`, `supplier_id/group`, `resource_group`, `source_type/source_id`, а спорные совпадения отправлять в `needs_review`.
- Нельзя физически удалять дубли поставщиков обычной миграцией. Дубли связывать логически через ИНН/ОГРН/email/телефон/нормализованное название/aliases и показывать предупреждения, если уверенность низкая.
- В общий `App.js` не класть бизнес-логику company-context, КП, поставщиков или сметного контроля. `App.js` может только подключить провайдер/пропсы; реализация должна жить в `src/features/company-context/`, `src/features/supply/`, `src/features/warehouse/`, `src/features/platform-admin/` и backend helper/services.
- Порядок внедрения: 1) документация и read-only диагностика текущих `company_id`, дублей поставщиков, невидимых КП и несопоставленных материалов; 2) `user_company_roles` и company-context без изменения рабочих цепочек; 3) переключатель компаний и запрет рабочих действий без выбранной компании; 4) запись `company_id` в новые заявки/КП; 5) `supply_request_recipients` и preflight видимости КП; 6) запись `company_id` в новые накладные, счета и складские движения; 7) backend endpoint кабинета поставщика по компаниям; 8) ресурсные группы материалов и мягкий сметный контроль; 9) backfill старых документов только после dry-run; 10) smoke/e2e и поэтапное выключение старых обходных путей.
- Обязательные проверки перед релизом: `system_owner -> создать platform_account -> две companies -> один директор имеет доступ к обеим -> режим Все компании только read-only -> заявка/КП создается только в выбранной компании -> поставщик видит запрос в своем кабинете по компании-клиенту -> накладная и счет не дублируют склад/оплату -> старые документы остаются доступны`.
- Решение от 2026-07-10: полноценный живой E2E `один директор -> компания А + компания Б` пока отложен. Для него еще нет безопасной тестовой инфраструктуры: двух тестовых компаний одного `platform_account`, отдельных объектов, членств/ролей, рабочего переключателя контекста и гарантированной очистки тестовых документов. Текущую production-базу нельзя использовать для имитации этого сценария вручную.
- До создания второй тестовой компании сначала закрыть дыры действующей ERP: завершить оставшиеся write/read-срезы склада (`M4.9-M4.11`), финансы и бухгалтерию (`M5`), проекты/сметы/ЖПР/акты/файлы/уведомления/AI (`M6`), затем выполнить read-only dry-run старых данных и только после него backfill/ограничения (`M7`). Не начинать UI и E2E двух компаний, пока эти контуры могут читать или писать строки без проверенного `company_id`.
- Подтверждено production smoke 2026-07-10 на версии `e7f92466ca86`: базовый и директорский smoke прошли без API-ошибок; полная цепочка `сметный материал -> заявка -> получатель КП -> КП -> счет -> поставка -> складская накладная -> материалы/история` прошла и очистила тестовые записи. Проверено сохранение одной компании по связанным документам, но это не является отрицательным тестом невидимости данных компании А в компании Б.
- Условие старта двухкомпанейного E2E: все M4-M6 write/read-контуры закрыты; есть отдельный `smoke:multi-company`; он создает только временные компании/объекты/документы, проверяет запрет записи в `Все компании`, невидимость А из Б и Б из А, одинаковый `company_id` по всей цепочке, а затем гарантированно очищает тестовые данные. До выполнения этих условий checkpoint SaaS Boundary остается открытым.
- Утвержденный порядок `M6`: `M6.0` реестр tenant-таблиц/маршрутов и источников компании; `M6.1` проекты как корень доступа; `M6.2` защищенные файлы и версии документов; `M6.3` сотрудники, членства и персональные документы; `M6.4` сметы, версии, изменения и чат; `M6.5` ЖПР, помещения, договорные позиции и акты; `M6.6` поручения, AI/OCR и фоновые задачи; `M6.7` MAX, уведомления и outbox; `M6.8` audit, экспорты и отрицательные тесты. Не менять порядок без отдельного архитектурного решения.
- Критические правила `M6`: `project_name` не является границей безопасности; новые дочерние записи наследуют immutable `company_id` и `project_id` от проверенного родителя; прямой маршрут по ID проверяет сохраненную компанию записи; режим `Все компании` запрещает любые изменения; финансовый факт остается в `M5`, а содержание/подписи акта - в `M6`.

Постоянный публичный URL не является допустимой моделью для накладных, актов и персональных документов.
- В снабжении не делать многоуровневое дерево заявок. Внутри существующих вкладок нужен один простой срез по источнику: `Все / Из сметы / Вручную / Проверка`; заявки `Из сметы` группировать одной строкой по объекту и сметному пакету (`Общестрой`, `Электрика`, `Слаботочка` и т.д.), а ручные заявки мастеров/прорабов/директора держать отдельно. Это только UI-группировка, она не создает вторую цепочку КП/поставки/склада.

### Авторизация и сессионная безопасность

- Долгосрочная цель: не хранить основной `authToken` входа в `localStorage`; перевести авторизацию на серверную сессию или refresh-cookie с флагами `HttpOnly`, `Secure`, `SameSite=Lax/Strict`.
- `localStorage` использовать только для неопасных UI-настроек и локальных черновиков. Токены, reset-коды, 2FA-секреты, персональные данные и документы не должны храниться там как источник правды.
- Если Bearer-token временно остается во frontend, он должен быть короткоживущим, с серверной проверкой срока действия, нормальным logout/invalidate и без хранения чувствительных данных в payload.
- Любая миграция авторизации должна сохранить роли, 2FA, audit log, кабинет платформы и клиентские кабинеты; сначала добавить совместимый серверный слой, потом постепенно убрать чтение `authToken` из `localStorage`.
- Для защиты от XSS добавить и поддерживать CSP-заголовки, запрет небезопасного HTML, аккуратную обработку пользовательских файлов/распознавания документов и отдельные smoke-проверки авторизации.
- `/parse-smeta` не является публичным endpoint: импорт сметы требует авторизованного пользователя и принимает только Excel-форматы openpyxl с лимитом размера до разбора файла.
- ✅ Начат первый совместимый слой: `user_sessions` + `HttpOnly` session-cookie рядом со старым Bearer `authToken`, `/logout` инвалидирует cookie-сессию, а `get_current_user` принимает оба механизма. Старый `localStorage.authToken` пока оставлен как fallback до отдельного frontend-перехода.
- ✅ Слой сессий задеплоен на production и проверен `npm run smoke:auth-session`: логин возвращает переходный Bearer-token, ставит `HttpOnly` cookie, защищенный endpoint открывается по cookie, `/logout` отзывает cookie-сессию, старый Bearer fallback остается совместимым.
- ✅ Frontend-запросы переведены в cookie-first режим: общий fetch-wrapper сначала отправляет `credentials: 'include'` без Bearer, а старый `localStorage.authToken` использует только как совместимый fallback после `401`.
- ✅ CSRF совместимый слой: `/csrf-token` выдает подписанный токен, привязанный к текущей `HttpOnly` cookie-сессии; frontend добавляет `X-CSRF-Token` для защищенных изменяющих запросов; первый enforcement включен на production только для cookie-logout через `CSRF_LOGOUT_ENFORCED=true`. Публичные `/login`, `/register` и password-reset endpoints намеренно не требуют CSRF. При добавлении новых top-level auth endpoints не забывать добавлять их в nginx API whitelist, иначе nginx отдаст `index.html` вместо JSON.
- ✅ Отзыв сессий при критичных изменениях пользователя: при отключении через `PUT /users/{id}` с `active:false`, смене пароля, смене роли через `PUT /users/{id}`, сбросе 2FA через `POST /users/{id}/2fa-reset` и отключении через staff-карточку `DELETE /staff/{id}` активные `user_sessions` пользователя помечаются `revoked_at`, чтобы старые cookie-входы не оставались живыми. Следующий отдельный шаг: держать совместимость Bearer fallback 1-2 релиза и после живых проверок постепенно ужесточать старый token-путь.
- Не удалять старый Bearer-token одним большим изменением: сначала 1-2 релиза держать совместимый режим, проверять роли, 2FA, клиентский кабинет, мобильную загрузку смет/склада/ЖПР и только потом сокращать TTL/убирать `authToken` из frontend-хранилища.

### Полноценное приложение

- Не переписывать ERP с нуля в native на первом этапе. Базовый путь: довести мобильный web/PWA до стабильного состояния, затем упаковать текущий React frontend в Capacitor-приложение для iOS/Android с нативными возможностями.
- MVP приложения должен закрывать полевые сценарии, а не весь кабинет директора: вход/2FA, объекты, сметы только в легком списке, ЖПР, фото/файлы, накладные, склад объекта, заявки, уведомления, офлайн-черновики и повторную отправку при плохой связи.
- Нативные функции первого релиза: камера/галерея, загрузка PDF/фото, push-уведомления, безопасное хранение сессии, базовый offline-cache для черновиков, deep-links на объект/задачу/смету.
- Desktop/admin-функции не тащить целиком в первый mobile app: настройки платформы, бухгалтерские сверки, массовые справочники, тяжелые таблицы смет и аналитика остаются web-first до отдельного мобильного дизайна.
- Реалистичная этапность: 1-2 недели - стабилизировать mobile web/PWA и auth cookie; 3-6 недель - Capacitor MVP с камерой, файлами, push и тестами на Android/iOS; 2-3 месяца - рабочий полевой релиз с офлайн-черновиками, role-based QA и публикацией; 4-6+ месяцев - полноценный SaaS-app со всеми ролями, если не переписывать все с нуля. Полный native rewrite может занять 6-12 месяцев и сейчас не является рациональным первым шагом.

- Backend-расширения CRM размещать в `backend/features/crm/`; в `backend/main.py` оставлять только регистрацию модуля и совместимые старые endpoints.
- Frontend CRM размещать в `src/features/crm/`; `src/App.js` должен только загружать модуль, передавать сервисные props и обновлять список.
- CRM является входным шлюзом внешних заявок: клиенты, поставщики, мастера, бригадиры, субподрядчики и партнеры сначала попадают в CRM на проверку.
- Заявка из CRM не должна автоматически выдавать роль или доступ. Доступ появляется только после ручного одобрения, создания карточки поставщика/исполнителя и приглашения.
- Паспортные данные, банковские реквизиты, договоры и документы сделки хранить в карточке CRM/сделки, но не показывать чувствительные поля в списках и канбане.
- Распознавание документов контрагента делать через общий модуль: поставщик, мастер, бригадир, субподрядчик или клиент загружает реквизиты/паспорт/договор, ИИ готовит черновик полей CRM и предмет договора, но карточка и доступ меняются только после ручного применения и сохранения.
- После подписания договора клиентская сделка превращается в объект, а документы и сметы должны связываться с существующими проектными документами и сметным модулем, а не дублироваться отдельно.

### ERPNext как ориентир, а не замена

- ERPNext использовать как карту зрелой ERP-логики: справочники, карточки, статусы и связи между accounting, procurement, warehouse, projects, CRM и HR.
- Не переходить на ERPNext и не копировать его экраны целиком: Stroyka остается строительной ERP вокруг объекта, сметы, пакета работ, ЖПР, АОСР, склада объекта, фото первички и ролей стройки.
- Полезные сущности для ориентира: `Материал/Item`, `Поставщик/Supplier`, `Единица/UOM`, `Склад/Warehouse`, `Заявка/Material Request`, `КП/Supplier Quotation`, `Счет поставщика/Purchase Invoice`, `Складское движение/Stock Entry`, `Оплата/Payment`, `Лид/Lead`, `Сотрудник/Employee`.
- Любое ERPNext-подобное улучшение внедрять только через существующие цепочки Stroyka, без параллельных экранов и без второго источника правды.
- Смета заказчика, пакет работ, объектный склад, ЖПР и фактические оплаты в `project_payments` остаются главными источниками правды для строительной экономики.

### Безопасное внедрение ERP-цепочек

- Перед изменением справочников, закупок, склада, бухгалтерии, CRM или HR обязательно описать, какие цепочки затрагиваются:
  `смета/нормы -> материалы -> заявка -> КП -> поставка -> накладная -> склад -> выдача -> списание -> бухгалтерия`;
  `счет поставщика -> складская накладная -> бухгалтерская проверка -> проектная оплата`;
  `объект -> пакет -> исполнитель -> ЖПР -> акт -> оплата -> удержание`;
  `лид -> клиент -> объект -> смета -> договор/работы`;
  `сотрудник/поставщик -> роль/доступ -> документы -> действия в audit_log`.
- Внедрять по слоям: сначала справочники и алиасы, затем связи документов, затем складовые движения, затем бухгалтерские статусы и только после этого аналитика/рейтинги/автоматизация.
- Первые версии новых цепочек должны работать в режиме предпросмотра или `needs_review`, пока реальные документы не подтвердят корректность сумм, НДС, единиц, поставщика, объекта и пакета.
- Нельзя менять остатки склада без документа-основания: приход, перемещение, выдача, списание, сторно или корректировка должны оставлять запись в истории.
- Нельзя считать счет, накладную и оплату тремя расходами: счет и накладная являются основанием/долгом, фактические деньги появляются только через `project_payments`.
- Нельзя автоматически удалять или схлопывать старые поставки, накладные, материалы и поставщиков при нормализации справочников; дубли связывать и скрывать в интерфейсе только после проверки.
- Нельзя давать ERP-улучшению обходить проектный и пакетный доступ: прораб, мастер, бригадир, субподрядчик, снабженец, кладовщик и бухгалтер должны видеть только разрешенные объекты, склады и пакеты.
- Если новое правило не может уверенно сопоставить материал, поставщика, единицу, пакет или оплату, оно должно оставить документ на проверку, а не молча проводить его дальше.

### OpenProject как ориентир для карты руководителя

- OpenProject использовать как ориентир по объектному графику: пакеты работ, этапы, Gantt, сроки, стоимость, ответственные, командная работа и связь задач с фактами.
- Не заменять главную страницу и не превращать карту в отдельную ERP. В Stroyka это отдельная вкладка объекта `Карта руководителя` рядом с `График` и `Этапы`.
- Первый слой карты должен быть read-only: показывать связи, риски и переходы к источникам, но не менять склад, ЖПР, АОСР, оплаты, сроки и статусы.
- Не вводить жесткие блокировки. Все сомнительные связи, нехватка данных и низкая уверенность идут в состояние `needs_review`, а не останавливают работу пользователей.
- Источники карты: активные сметы заказчика и пакет работ, `project_stages`, ЖПР, контроль материалов, заявки снабжения, поставки, счета поставщиков, складские накладные, входной контроль, АОСР и `project_payments`.
- Факт денег считать только из `project_payments`. Счета поставщиков и накладные показывать как обязательства или первичку, чтобы не получить двойной расход.
- Если карта открывает действие, она должна вести пользователя в существующую вкладку-источник: материалы, ЖПР, АОСР, входной контроль, финансы, смету или ИИ-контроль.
- Мобильная версия карты должна быть компактной: KPI в 2 колонки, фильтры без горизонтальной обрезки, таблицы связей превращать в карточки.

### Объект, сметы и пакеты

- У объекта может быть несколько пакетов: `общестрой`, `отделка`, `электрика`, `слаботочка`, `отопление`, `вентиляция`, `ВК/канализация`, `благоустройство` и другие.
- Базовая единица исполнения - `объект + пакет работ`, а не вся смета и не старый наряд.
- Заказчик видит только активные сметы типа `Заказчик` по своему объекту.
- Список смет показывается по объектам: сначала видны только заголовки объектов с количеством папок/статусов/суммой, а пакеты и версии раскрываются по клику на объект.
- Исполнитель получает доступ только к назначенным объектам и пакетам.
- Мастер, бригадир и субподрядчик не видят чужие сметы, чужие пакеты и чужие расценки.
- Цена заказчика и цена исполнителя хранятся отдельно.
- Основной сценарий выдачи работ исполнителю: `Назначить мастеру` в карточке сметы. Пользователь выбирает мастера/бригаду, строки работ и коэффициент или ручную цену, а система создает/обновляет существующие `brigade_contracts` и `brigade_contract_items`. Старое `Распределить` остается как расширенный режим, но не должно быть главным путем для обычной выдачи работ.
- В карточке сметы должен быть виден реестр выдачи работ: сколько строк назначено, какие строки не выданы, кому назначены позиции, план/выполнено по исполнителю. Ошибочное назначение можно снять из этого реестра только пока по строке нет выполненного объема.
- Парсер смет не должен терять рабочие строки в основной смете только из-за материальных слов в названии. Если импортированная строка явно пришла как `work` и рабочая сумма находится в `totalWork`, она остается работой даже при словах `светильник`, `грунтовка`, `штукатурка`, `плитка`; ресурсной строкой она становится только при ресурсном коде/типе или материальной сумме.
- Материальные строки сметы с количеством `< 0` или суммой `< 0` считаются корректировкой: они учитываются в общей сумме сметы, но не попадают в материалы, заявки, потребность, остатки и списание.
- Строки с количеством `0` не формируют заявку и списание, пока нет реального количества.
- При загрузке новой активной редакции сметы того же объекта, типа и пакета создаётся `Сверка смет`: снимок старой/новой сметы, спорные позиции, связь с утверждёнными изменениями к смете и запись в реестре документов объекта.
- Утверждённая сверка смет показывает, какие изменения закрыты новой сметой, а какие остаются отдельной допработой для ДС/КС. Включённые в новую смету изменения не должны повторно попадать в КС отдельными строками.
- Сверка смет хранится отдельно от живых строк сметы: последующее редактирование сметы не должно менять уже созданный документ сверки.

### Дневные акты исполнителей и допработы

Записка 2026-07-07:

- В кабинете мастера, бригадира и субподрядчика выбранные сметные позиции за одну дату должны собираться в один дневной акт выполнения по связке `объект + пакет + исполнитель + дата`, а не в разрозненные ручные записи.
- Дневной акт создается из уже назначенных строк сметы/договора исполнителя, подтягивает объемы, помещение/зону, единицы, цену исполнителя, фото и комментарии. Фото работ прикладываются к акту и к факту ЖПР, чтобы бухгалтерия и директор видели одно основание.
- Если исполнитель за день закрывает несколько позиций, они попадают в один акт с несколькими строками. Отдельный акт нужен только при другом объекте, пакете, исполнителе или дате.
- В интерфейсе исполнитель вводит не накопленное `итого сделано`, а дневной объем по строкам. Кнопка `Рассчитать день`/`Сформировать работы в один акт` открывает предпросмотр: строки, объем за день, сумма исполнителя, дата, общий фотоотчет и комментарий проверяющему. После подтверждения система отправляет все выбранные строки одним пакетом в ЖПР со статусом `На проверке`.
- Серверная автосклейка дневного акта после подтверждения ЖПР остается контрольным дневным пакетом: подтвержденные строки по связке `объект + пакет + исполнитель + дата` агрегируются в `interim_act` с `source_type='daily_work'`, но этот пакет не оплачивается напрямую.
- Оплатный/печатный акт подрядчику формируется бухгалтерией вручную из вкладки `Акты`: выбирается исполнитель, объект, пакет и период либо режим `за всё время работ`. В акт попадают подтвержденные ЖПР этого подрядчика, которые еще не включены в финальный акт; дневные контрольные пакеты не считаются дублем. Печатная форма акта должна брать состав строго из `work_journal_ids`, чтобы сметчик не собирал строки вручную.
- Ручной ввод работ без сметной позиции оставить только как черновик `Работа вне сметы`/`Изменение к смете`; он не должен сразу попадать в акт, оплату или КС без подтверждения директора/сметчика.
- Поток допработ: исполнитель/прораб заводит черновик -> директор/сметчик подтверждает тип (`изменение к смете`, `работа вне сметы`, `ошибка сметы`, `расходник/комплектация`) -> после подтверждения строка попадает в отдельный реестр допработ, ДС/КС и актирование.
- При новой редакции сметы подтвержденные изменения нужно сверять с `Сверкой смет`: если изменение уже включено в новую смету, оно не должно повторно закрываться как отдельная допработа.
- Вкладку `Работы` у мастера/бригадира/субподрядчика нужно профилировать отдельно: сейчас критичный риск - большие сметы и расчеты материалов/остатков/подсказок внутри рендера. Оптимизация должна сначала кэшировать список назначенных работ, материалы по объекту/пакету и подсказки норм, а не менять бизнес-цепочку.

### Свод одинаковых работ

- В карточке сметы есть `Свод одинаковых работ`.
- Свод группирует одинаковые работы по объекту, активной смете/пакету, нормализованному названию, единице и типу обмера.
- Свод не заменяет исходную смету и не переписывает импорт.
- В интерфейсе сметы свод показывается компактной таблицей только по повторяющимся работам; источники строк раскрываются внутри строки, без больших карточек и отдельного широкого экрана.
- В своде должны храниться ссылки на исходные строки: `estimateId`, версия, раздел, номер/индекс строки, исходное название, объем, единица, цена, сумма.
- Директор или зам назначает цену исполнителю на агрегированную работу; система применяет ее к связанным исходным строкам для договоров, ЖПР, актов и расчетов.
- Исполнитель видит только свою работу, доступный объем, помещения/зоны и свою цену/заработок. Сметные суммы заказчика не показывать.
- Работы с разными единицами, разным пакетом или спорной логикой обмера не объединять автоматически; выводить как требующие проверки.

### Помещения, зоны и факт выполнения

- Если объект ведется по помещениям, факт закрывается по связке `объект + пакет + помещение + работа + исполнитель`.
- Помещение выбирается из справочника объекта, не вводится свободным текстом.
- Для фасада, кровли, наружных сетей и благоустройства вместо помещения используется зона/участок/фронт работ.
- Окна, двери и откосы участвуют в обмерах: их нельзя просто выкидывать, потому что они могут быть отдельными работами.
- Один и тот же объем одной работы в одном помещении не должен закрываться двумя исполнителями без явного разбиения по объему, этапу, зоне или операции.
- Из одного факта выполнения система ведет ЖПР, профильные журналы, историю помещения/зоны, кандидатов в АОСР, акты исполнителя и аналитику.

### Материалы, склад и накладные

Материал проходит путь:

`смета/норма -> потребность -> заявка -> КП -> поставка -> приход -> склад объекта -> выдача исполнителю -> списание через ЖПР`

Правила:

- Нормы расхода не заменяют физическую выдачу материала.
- Исполнитель не должен списывать материал, который ему не выдали.
- Выдача материала исполнителю привязана к объекту, пакету и конкретному пользователю с ролью `мастер`, `бригадир` или `субподрядчик`.
- Перед выдачей сервер повторно проверяет сметный контроль по объекту и пакету.
- Основной приход материала идет через `заявка -> КП -> отгрузка -> приемка`.
- Ручная приходная накладная на объект разрешена для прямого заказа директора/зама или материала вне цепочки снабжения.
- Ручная объектная накладная должна сохраняться как `sourceType=manual_project_invoice`, иметь объект, пакет, поставщика, номер/дату, позиции, фото и складскую историю.
- Ручная накладная не должна попадать в `Мои траты`.
- Прораб может принимать ручную, Telegram- или MAX-накладную только на закрепленный объектный склад. Основной склад прорабу закрыт.
- Быстрое действие `Принять на склад` всегда создает складскую накладную, а не личную трату.
- Любая принятая складская накладная должна автоматически создавать или связывать бухгалтерскую первичку поставщика (`supplier_invoices`) и обратную ссылку `warehouse_invoices.supplier_invoice_id`, даже если приход сделан вручную, через веб, MAX mini-app или webhook. Оплату не проводить автоматически.
- Telegram для складской накладной должен использовать `/telegram/warehouse-invoices`; `/telegram/own-expenses` только для личных трат и должен отклонять складские payload.
- MAX для складской накладной должен использовать `/max/warehouse-invoices` через общий слой мессенджеров. Он не проводит отдельный склад, а вызывает ту же серверную функцию создания складской накладной, что веб и Telegram.
- Название материала в накладной может отличаться от сметы. Сервер применяет алиасы, затем безопасное fuzzy-сопоставление по названию, единице и пакету.
- Fuzzy-сопоставление материалов должно покрывать типовые различия накладных и ГРАНД-смет: `гипрок` = `лист гипсокартонный`, `ПН/ПП/CD/UD/UW/CW` = профиль/направляющая, `кабель` = `провод` внутри кабельного семейства, `ППР/полипропилен` = труба, `LED/прожектор` = светильник, `самонарезающий шуруп/прессшайба` = саморез/крепеж. Упаковка, цвет и бренд не должны быть главным ключом.
- При уверенном совпадении со сметной ресурсной строкой складская строка прихода хранится под каноническим сметным названием/пакетом, а исходное название из документа сохраняется в `invoiceOriginalName`.
- Если материал найден в другом активном пакете объекта, он должен лечь в найденный пакет, а не зависать в `Основная`.
- Приход, перемещение и выдача ищут складскую строку по точному названию, алиасу и безопасному похожему названию с совместимой единицей.
- Передача материалов исполнителю поддерживает несколько выбранных складских строк за один проход формы: каждая строка сохраняет свое количество, единицу и пакет, а выбор следующего материала или получателя не сбрасывает уже выбранные строки.
- Сканированная объектная накладная должна приходоваться даже при предупреждениях сметного контроля: бренд/фирма не должен быть главным ключом, материал сопоставляется по семейству и словам полного наименования, расходники (`валик`, `шпатель`, `нож` и т.п.) фиксируются как `Расходник вне сметы`, а не блокируют склад.
- Скан счета/накладной может быть неполным фрагментом страницы: если видна товарная таблица, документ все равно распознается и приходуется; отсутствующий номер заменяется черновым `SCAN-YYYYMMDD-HHMM` с предупреждением в предпросмотре.
- Для повторяющихся поставщиков используется слой шаблонов по принципу `invoice2data`: скан сохраняет исправления предпросмотра как правило поставщика, будущий AI/OCR получает подсказку по этому поставщику и отмечает результат как `template`; если правило не найдено или распознавание сомнительное, документ остается в обычном AI/OCR-предпросмотре.
- Если материал из накладной относится к укрупненной работе сметы, но не выделен отдельной ресурсной строкой, он фиксируется как `Комплектация укрупненной работы`.
- Материал без сметного соответствия и без связи с укрупненной работой считается `вне сметы` и требует решения: доп. смета, расходник, укрупненная работа или ошибка распознавания.
- Себестоимость материалов объекта считать по фактическому приходу, а не по текущему остатку.

### Журналы материалов, кабеля и замкнутый цикл прихода

- Целевой цикл: `поставка/накладная -> склад объекта -> входной контроль -> кабельный журнал -> сверка со сметой -> выдача -> списание через ЖПР`.
- Реализация 2026-07-03: базовый backend-backfill уже связывает приходы из `supply_deliveries`, `warehouse_invoices`, `warehouse_history` и `materials` с входным контролем; кабельный журнал восстанавливается из поставок, накладных, истории склада и остатков объекта; карточки журналов показывают `не связано`; smoke снабжения проверяет входной контроль, кабельную накладную и отсутствие дублей при повторном открытии журналов.
- `Журналы объекта` не должны быть просто счетчиками отдельных таблиц. Они должны показывать факт по источникам: сколько пришло на склад, сколько попало во входной контроль, сколько попало в кабельный журнал, сколько осталось `не связано`.
- Любой приход на объект из `supply_deliveries`, `warehouse_invoices`, `warehouse_history` или `materials` должен иметь связанную запись входного контроля. Если записи нет, backend должен восстановить ее через backfill без ручного создания пользователем.
- Кабельный журнал должен строиться не только из уже созданного `cable_journal`, но и из фактических кабельных строк склада/накладных/поставок. Если кабель есть на складе объекта, но не попал в журнал, система должна показывать это как `кабель без журнальной записи`.
- Распознавание кабельной продукции нужно расширять максимально широко и безопасно: силовой кабель (`ВВГ`, `ВВГнг`, `ВВГнг-LS`, `NYM`, `КГ`, `КВВГ`, `ПВС`, `ШВВП`, `ПуГВ`, `ПВ-1`, `ПВ-3`, `СИП`, `АВВГ`), слаботочный/СКС (`UTP`, `FTP`, `F/UTP`, `U/UTP`, `S/FTP`, `SFTP`, `витая пара`, `Cat5e`, `Cat6`, `коаксиальный`, `RG-6`), пожарный и охранный (`КПС`, `КПСЭ`, `КПСВВ`, `КСВВ`, `КСПВ`, `КСПЭВ`, `FRLS`, `FRHF`, `нг(А)-LS`, `ОПС`), связь/домофон/сигнализация (`ТППэп`, `КВПЭф`, `J-Y(St)Y`, `JE-H(St)H`). Список должен жить в одном backend-справочнике/утилите, а не дублироваться по экрану.
- При распознавании кабеля нельзя путать кабель с комплектующими трассы: `гофра`, `кабель-канал`, `лоток`, `короб`, `клипса`, `скоба`, `стяжка`, `муфта`, `наконечник`, `сальник` - это отдельные материалы, но не строки кабельного журнала.
- Для кабеля сохранять исходное название из накладной, каноническое название, тип системы, жилы, сечение, длину, единицу, поставщика, дату, объект, пакет, источник (`invoice_id`, `delivery_id`, `warehouse_history_id`) и статус проверки.
- Сверка пришедших материалов со сметой должна использовать общий механизм для активных смет `Заказчик` и `Материалы`, алиасов, единиц измерения и пакетов работ. Нельзя считать материал `вне сметы`, если он есть в смете под похожим названием, брендом, другой формой записи или в другом активном пакете того же объекта.
- Материал, который не удалось уверенно сопоставить, не блокирует приход. Он получает статус `вне сметы / требуется проверка` с вариантами решения: создать алиас, связать со сметной строкой, отнести к расходникам, комплектации укрупненной работы или доп. смете.
- На карточке объекта нужно показывать диагностические счетчики: `входной контроль: создано / не создано`, `кабель: на складе / в журнале / не распознано`, `сметная сверка: по смете / сверх / вне сметы / требует алиаса`.
- Внедрять без отдельной новой вкладки: использовать существующие `Журналы объекта`, `Входной контроль`, `Кабельная продукция`, `Материалы/Склад объекта` и карточку накладной.
- Следующий проход: довести отчет `не связано` до отдельного режима фильтра в существующих вкладках и расширить реальные алиасы по данным первых продовых накладных.

### Нормы материалов и AI

- Глобальные нормы компании не должны автоматически засоряться слабыми AI-предложениями.
- Слабые AI-предложения оставлять на проверку, не сохранять в справочник без подтверждения.
- AI-предложенную норму расхода можно исправить прямо в карточке предложения перед сохранением в справочник или как поправку объекта; сохраняется проверенное пользователем значение, а не слепой расчет AI.
- Результат `Вся смета по нормам` можно использовать для черновика доп. материалов или доп. работ, но активную смету заказчика не менять автоматически.
- Списание по нормам сервер принимает только если норма найдена и подтверждена в базе/правилах.
- План закупки строится только из самостоятельных ресурсных строк активной сметы. Нормативная потребность по работам остается контрольной подсказкой и не создает заявку, пока материал не подтвержден строкой сметы или решением директора/сметчика.
- В закупочном расчете разные материалы нельзя автоматически схлопывать только по общему семейству (`крепеж`, `кабель`, `профиль`, `клей`). Автоматическое объединение разрешено для точного канонического наименования или подтвержденного алиаса; похожие позиции показываются как кандидаты на ручную связь.
- AI-норма, построенная из конкретной работы, должна совпадать с полной сигнатурой работы, а не с одним общим словом. Совпадение `кладк` внутри `прокладка`, `провод` внутри `трубопровод` и `штукатур` в выражениях `по штукатурке`/`под штукатурку` не является достаточным основанием для расчета материала.
- Корректировочные расценки (`добавлять/исключать`, `на каждые 5 мм`) не создают отдельную нормативную потребность без связи с базовой работой и фактической толщиной.
- Поправка нормы объекта или сметы с `baseNormId` заменяет связанную базовую норму в своей области действия, а не складывается с ней.
- Любая сомнительная строка остается видимой в `Проверить смету/норму`, но исключается из `Докупить` и массового создания заявок до подтверждения.
- Статус P1 от 2026-07-10: закупочная сверка объединяет только точное каноническое наименование или подтвержденный алиас; складской прием остается на прежней неблокирующей логике. Устранены ложные нормы для `прокладка/кладка`, отделки по штукатурке, трубопроводов, пароизоляции и корректировок толщины стяжки; связанная поправка заменяет базовую норму. Этап не меняет сохраненные сметы, накладные и заявки.
- Статус P2 от 2026-07-10 (`e1f317ef`, production): сметный материал и нормативная подсказка разделены в колонках, фильтрах, итогах и печатной ведомости. Каждую строку можно раскрыть до сметы, пакета, раздела, работы, исходного количества, преобразования единицы, правила и формулы. Конфликт единиц, некорректная строка или неподтвержденная связь получают статус `Проверить`; такие позиции исключены из одиночного и массового создания заявок. Нормативные строки остаются подсказкой и сами не попадают в `Докупить`.

### Бухгалтерия, договоры и выплаты

- Бухгалтер получает реестр закрытия по исполнителю, объекту и пакету.
- Один субподрядчик может работать на нескольких объектах: карточка исполнителя общая, договоры/акты/платежи привязаны к объектам.
- Месячное закрытие собирается из подтвержденных объемов ЖПР и внутренних расценок.
- Удержание 5% на 1 год отражается в договоре, акте, оплате и остатке.
- `own_expenses` - реестр возмещения сотруднику; `expenses` - фактический расход.
- При создании `Моей траты` из веба или Telegram backend автоматически создает связанную строку в `expenses`.
- При отклонении траты связанный расход удаляется из экономики объекта; при возврате в рабочий/возмещенный статус создается снова.
- Быстрый `Расход по объекту` поддерживает вложения: фото с камеры, файлы/PDF/документы и несколько файлов за один расход; вложения видны в финансах объекта.
- Прямой `Расход по объекту` из `expenses` учитывается в себестоимости объекта, бухгалтерской сводке, платежах по объектам и блоке `Факт расходов`; строки `expenses`, созданные из `own_expenses`, не прибавлять второй раз к возмещениям.
- Факт оплаченных денег считать из `project_payments`; счета и акты - основание/долг, их нельзя повторно прибавлять как факт оплаты.

### Бухгалтерская первичка и входящие документы

- Каждая складская накладная после сохранения попадает в бухгалтерскую очередь входящих документов как первичный документ, а не как строка материала.
- Бухгалтерская сводка показывает очереди `Нет фото`, `На проверке`, `Нужно уточнение`, `К оплате`, `Частично оплачена`, `Оплачена`, `Отклонена`.
- Накладная без фото получает бухгалтерский статус `Нет фото`; ее можно открыть и дозагрузить фото к уже созданному документу.
- Складской статус накладной (`Принята`, `Аннулирована`) нельзя подменять бухгалтерским статусом; для бухгалтерии используются отдельные поля `accounting_status`, `accounting_comment`, `paid_amount`.
- Строки вне сметы, расходники и комплектация укрупненных работ не блокируют складской приход, но оставляют документ в бухгалтерской проверке до решения.
- `К оплате` означает, что бухгалтер принял первичку как основание. Фактический расход появляется только после оплаты через `project_payments`.
- Счёт поставщика и складская накладная должны связываться в комплект первички: автоматически по `offer_id/request_id`, вручную во входящих документах бухгалтерии, если документы пришли раздельно.
- Оплата связанного документа синхронизирует бухгалтерский статус складской накладной и статус/оплаченную сумму счёта поставщика, но не создаёт второй факт расхода помимо `project_payments`.
- В бухгалтерии нужна отдельная вкладка `Документы от поставщиков`: список группируется по поставщику, внутри карточки поставщика показываются все связанные счета/УПД/складские накладные, даты, суммы, объект, статус сверки, оплата и расхождения. Из строки документа бухгалтер должен открыть фото/файл, распознанные позиции, складскую накладную, связанную первичку и историю оплаты.
- Вкладка `Документы от поставщиков` нужна для сверки с поставщиком: быстро увидеть пропущенные накладные, не связанные счета, расхождения по сумме/НДС/датам и документы без фото. Спорные документы переводить в `Нужно уточнение`, а не скрывать.
- Поставщик - единая карточка в снабжении, а не строка на каждую накладную. При приходе/скане накладной система сначала ищет поставщика по сильным реквизитам (`ИНН/ОГРН/email/телефон`) или явной alias-связи, затем связывает с ним складскую накладную, счёт поставщика, поставку, КП и каталог; совпадение только по названию отправлять в `Нужно уточнение`, а не склеивать автоматически.
- Если при приходе накладной поставщика еще нет в базе, система создает карточку поставщика автоматически в статусе проверки и связывает с ней накладную/первичку. Когда поставщик позже регистрируется, карточка пользователя должна соединиться с уже созданной карточкой по ИНН/ОГРН/email/телефону или ручной связке дубля, без потери старых накладных, КП и оплат.
- Старые складские накладные без привязанного поставщика или `supplier_invoice_id` нужно обрабатывать отдельной сверкой/backfill: найти поставщика по ИНН/названию, привязать накладную и бухгалтерскую первичку к единой карточке, а после регистрации поставщика дать ему видеть эту историческую цепочку в кабинете.
- Кабинет поставщика не должен фильтровать заявки, КП, каталог, отгрузки, документы и претензии только по одному `supplier_id`: сначала строится единая группа поставщика по `user_id`, email, телефону, ИНН/ОГРН и явным `supplier_aliases`, затем показываются все связанные записи по всем id этой группы.
- Backfill старых накладных должен быть идемпотентным: не создавать повторную оплату, не дублировать складской приход и не плодить второй счет, если связь уже есть. Он должен пройти сверку, проставить связи и статусы, а спорные совпадения оставить на ручную проверку с понятной причиной.
- Нормализация поставщика отбрасывает ИНН/КПП/ОГРН, телефон, адресный хвост и организационную форму, чтобы `ООО ПКФ "ЭВЕРЕСТ"` и строка из счета с реквизитами попадали в одну карточку.
- Печатные формы договоров в бухгалтерии не должны ломать раздел: если старый/неполный договор не собирается, показывать понятное сообщение в предпросмотре, а не черный экран.

### Роли и доступ

- Персонал показывается группами: руководство, ИТР/прорабы, мастера/бригадиры, субподрядчики, снабжение/склад, бухгалтерия, заказчик/технадзор.
- Группы персонала на экране сотрудников должны сворачиваться и раскрываться по заголовку роли; по умолчанию список сотрудников не должен распирать экран.
- На телефоне персонал показывается карточками по этим группам, без широкой таблицы. Поиск должен находить по ФИО, роли, специализации, объекту, телефону, email, бригаде и категории.
- ✅ Мастера, бригадиры и субподрядчики внутри своих групп дополнительно раскладываются по пакету/специализации/бригаде, чтобы сразу видеть отделочников, электриков, сантехников, вентиляцию и другие направления.
- Внутри мастеров, бригадиров и субподрядчиков обязательна специализация/пакет: общестрой, отделка, электрика, слаботочка, отопление, вентиляция, ВК/канализация, благоустройство или другое.
- Исполнитель может иметь доступ к нескольким объектам и нескольким пакетам, но не видит чужие сметы, ставки и закрытые работы.
- Регистрация персонала и поставщиков идет через заявку/приглашение с подтверждением директора или замдиректора.
- Поставщик после регистрации получает статус `на проверке` и становится активным только после подтверждения.
- Закрытые аукционы делать позже, когда будет база исполнителей. Сейчас заложить структуру, но не превращать это в отдельный продукт.

### График руководителя

График объекта строится из активных смет, пакетов, обмеров, назначенных исполнителей, ЖПР, АОСР, поставок, перемещений, выдачи материалов, дедлайна и фактического прогресса.

Руководителю показывать:

- план/факт по пакетам;
- отставание;
- нехватку материалов;
- работы без помещения/зоны;
- неподтвержденные объемы;
- риски по смете и нормам.

### Мобильная PWA и скорость

- На телефоне интерфейс должен оставаться мобильным даже в горизонтальной ориентации.
- Склад, материалы, накладные, CRM, уведомления и быстрые действия показывать карточками/адаптивными сетками, без горизонтального уезда колонок.
- Большие сметы и склады не рендерить целиком по умолчанию.
- Первый экран сметы: объект, пакеты, свод работ, свод материалов. Исходные строки открывать по требованию.
- Тяжелые расчеты по сметам, материалам, нормам и сводам выполнять на backend и кешировать.
- Для больших таблиц использовать server-side поиск, `limit/offset` или курсор.
- Тяжелые разделы грузить лениво: сметы, склад, бухгалтерия, CRM, AI-контроль, печать.

### Стандарт производительности для новых функций

Любая новая функция Stroyka должна проектироваться с учетом будущего объема данных, а не только тестового объекта. Если функция может создать большой список, тяжелые счетчики, массовую загрузку документов или повторный пересчет в интерфейсе, сначала описать performance-подход и только потом писать код.

Обязательный чек-лист перед внедрением:

- `role-scope`: определить, какие роли видят функцию и какие объекты/пакеты/склады им доступны.
- `data-scope`: определить минимальный набор данных для первого открытия экрана; не грузить весь backend-пакет ради одной вкладки.
- `summary`: счетчики, суммы, долги, остатки, статусы очередей и dashboard-цифры считать на backend или через заранее подготовленный агрегат, а не через полный перебор больших массивов в React.
- `lazy-load`: тяжелые вкладки, детали карточек, историю, фото, позиции накладных, строки смет, документы и аудит грузить только по открытию или кнопке `Показать еще`.
- `list-limit`: любой список, который может вырасти больше 50-100 строк, должен иметь пагинацию, cursor/limit, виртуализацию или явную подгрузку; запрещено молча рендерить сотни строк при первом открытии.
- `no-full-loadAll`: после одного действия обновлять только затронутый домен данных, если это возможно. Полный `loadAll()` использовать только как совместимый fallback или ручное `Обновить`, потому что он перегружает сервер и интерфейс.
- `background-jobs`: OCR, AI-разбор, PDF, массовые сверки, backfill, уведомления и аналитические пересчеты делать через фоновые задачи/очередь или отдельный endpoint статуса, а не блокировать пользовательский клик.
- `indexes`: для новых backend-фильтров сразу предусматривать индексы по `company_id`, `project`, `project_id`, `supplier_id`, `user_id`, `status`, `date`, `work_package` или другому ключу выборки.
- `smoke`: новая цепочка должна иметь профильную проверку или расширение существующего smoke; для экранов с большой загрузкой дополнительно проверять отсутствие белого/черного экрана после входа.

Запрещено добавлять новый экран или крупную вкладку, если она:

- тянет все сметы, склад, накладные, поставщиков, ЖПР или бухгалтерию без фильтра роли и объекта;
- считает большие карточки через вложенные `map/filter/reduce` на каждый render;
- открывает сотни DOM-строк без лимита;
- после каждого клика вызывает полный `loadAll()` без причины;
- смешивает рабочие данные разных ролей или компаний ради удобства фронта.

Разрешенный базовый паттерн для новых модулей:

`экран -> компактный список/summary -> открыть карточку -> lazy details -> действие -> точечное обновление данных -> audit/smoke`

Если для функции пока нет backend-агрегата, первый безопасный слой - frontend-лимит, `useMemo`, `Показать еще` и ленивые детали. Второй слой - backend summary endpoint и индексы. Нельзя оставлять тяжелый frontend-перебор как постоянную архитектуру.

### Multi-company: финансовый блок M5

- M5 начинается после отдельного аудита бухгалтерии. Текущий раздел является управленческими финансами и первичными документами, а не заменой регламентированного учета/1С.
- Жесткая граница денег — `company_id`. Групповой режим `all_companies` только агрегирует разрешенные компании для чтения; любое изменение требует одну выбранную компанию и эффективную финансовую роль в ней.
- Первый шаг M5.1: `company_requisites` хранит отдельные юридические и банковские реквизиты каждой компании. Запрещено глобальное `DELETE FROM company_requisites`; сохранение выполняется как `upsert` по выбранному `company_id`.
- В режиме `all_companies` нельзя подставлять случайные реквизиты в договор, акт, счет или печатную форму. Перед формированием юридического документа пользователь обязан выбрать конкретную компанию.
- Шаг M5.2: `project_payments` является единственным журналом фактического движения денег; рабочая строка обязана иметь подтвержденный `company_id`, а неподтвержденная legacy-строка хранится в карантине с `company_id=NULL` и не участвует в tenant-чтении. Чтение строится по эффективным ролям каждой компании; заказчик видит только положительные поступления по назначенным ему объектам.
- M5.2 развернут в production коммитом `5db2e496`: `/health` подтвердил эту версию и доступность БД, публичный smoke прошел. Authenticated smoke выбранной компании остается до первичной настройки 2FA тестовой учетной записи.
- Ручной платеж, сторно, оплата акта, бригады или складской накладной обязаны записывать ту же выбранную/исходную компанию. Удаление связанной оплаты бригады создает сторно и не удаляет исходную строку денежного журнала. Режим `all_companies` не разрешает создание или сторно платежа.
- Пока `interim_acts` не получил собственный `company_id`, автоматическая оплата акта разрешена только при однозначном названии объекта. Если такое название есть в нескольких компаниях, сервер отвечает `409`, а не угадывает владельца денег.
- Шаг M5.3a закрывает чтение `/brigade-contracts`, `/brigade-payments`, `/brigade-contract-items-all`, `/brigade-contract-items/{contract_id}` и `/brigade-acts` через `brigade_contracts.company_id`. Режим `all_companies` объединяет только разрешенные компании, а исполнитель дополнительно ограничен своим договором, объектами и пакетами.
- Шаг M5.3b1 добавляет `company_id` и явный `project_payment_id` в `brigade_payments`. Старые договоры получают компанию только по точному `project_id` или глобально уникальному названию объекта; неоднозначные строки остаются без подтвержденной компании, скрыты из tenant-чтения и требуют ручной сверки. Старые выплаты наследуют только подтвержденную компанию договора и связываются с денежной проводкой лишь при строгом соответствии один-к-одному.
- M5.3b1 также убирает временный fallback `project_payments.company_id=1`: старые строки общего/неизвестного объекта и строки с пустой/нечисловой суммой получают `company_scope_verified=false`, `company_id=NULL` и не участвуют в tenant-чтении до ручной сверки. Все текущие серверные места создания `project_payments` уже передают компанию явно; новая строка без компании блокируется ограничением БД.
- Создание и сторно выплаты авторизуются по сохраненной компании договора, требуют эффективную финансовую роль в ней и записывают ту же компанию в `project_payments`. Новая выплата всегда хранит ID своей денежной проводки; удаление без однозначной ссылки блокируется, а не ищет проводку по похожей сумме/дате/комментарию.
- Выплата и ее денежная проводка выполняются одной транзакцией. Договор блокируется на время расчета доступного остатка, поэтому два параллельных запроса не могут дважды оплатить один и тот же выполненный объем. Сумма меньше `0.01 ₽`, несовпадение компаний, `NULL`, `NaN/Infinity` или испорченные суммы блокируют операцию ответом `409/400`.
- Шаг M5.3b2 требует одну выбранную компанию и эффективную роль директора/замдиректора при создании договора. Сервер блокирует точный объект внутри этой компании, явно записывает ее `company_id` вместе с каноническими `project_id/project_name` и не полагается на default компании `1`.
- Редактирование и аннулирование договора авторизуются по сохраненной компании договора, а итоговый `UPDATE` дополнительно содержит одновременно `id` и `company_id`. `all_companies`, конфликт body/header и попытка изменить договор другой компании блокируются до записи; бухгалтер может только аннулировать договор по прежнему правилу роли.
- Исполнитель договора ищется только среди активных пользователей/ролей и карточек персонала выбранной компании. Совпавшие ID разных таблиц и неоднозначные ФИО не угадываются; карточка без системного пользователя остается договором по имени без случайной привязки. Доступ к объекту/пакету добавляется в `user_company_roles` этой компании, а legacy-поля `users` обновляются только для той же default-компании пользователя.
- M5.3b2 не закрывает загрузку позиций из прайса отдельной командой, ручные изменения позиций, акты и распределение сметы. Эти write-маршруты остаются M5.3b3; до него бригадная цепочка не готова к multi-company пилоту.
- ИИ-контекст может читать `project_payments` только разрешенных компаний. `expenses` пока не участвует в multi-company сводке директорского агента: сначала таблица расходов должна получить собственный `company_id`.
- Следующие согласованные риски M5: добавить `company_id` расходам/подотчету/зарплате/договорам и источникам автоматических оплат; заменить определение направления денег по тексту комментария; проводить оплату документа и денежную запись одной серверной транзакцией; заменить физическое удаление финансовых строк на сторно.

### 2.1. ERPNext-ориентир: безопасная система внедрения

- Не внедрять ERPNext как отдельную подсистему. Брать только структуру цепочек и карточек, встраивая ее в текущие модули Stroyka.
- Первый слой: привести справочники к единой логике без изменения поведения склада: поставщик, материал, единица измерения, алиасы названий поставщика, алиасы материалов, связь материала с пакетом работ и сметной строкой.
- Второй слой: усилить связи документов без автопроведения: заявка снабжения, КП, поставка, счет поставщика, складская накладная, бухгалтерская очередь, оплата. На этом этапе показывать связи и предупреждения, но не менять старые документы автоматически.
- Третий слой: строгий складовой журнал. Любой остаток должен объясняться документом: приход, перемещение, выдача исполнителю, списание через ЖПР, сторно накладной или корректировка с причиной.
- Четвертый слой: бухгалтерская дисциплина. Связывать счет поставщика и складскую накладную в комплект первички, но факт денег учитывать только через `project_payments`.
- Пятый слой: рейтинги и аналитика поставщиков. Считать сроки, расхождения по суммам/НДС, качество документов, частоту ручной проверки, отклонения по цене от сметы и рекламации.
- Шестой слой: проектные и CRM-шаблоны. Лид может создавать объект, объект может поднимать типовой набор этапов, задач, документов и контрольных точек, но смету и договоры не создавать без подтверждения.
- Перед закрытием каждого слоя проверить цепочки: роли и доступ, объект/пакет, смета и материалы, снабжение, складские накладные, бухгалтерская очередь, оплаты, ЖПР/списание, audit_log, мобильный сценарий.
- Минимальные проверки для ERP-изменений: `PYTHONPYCACHEPREFIX=/tmp/stroyka-pycache python3 -m py_compile backend/main.py`, `npm run build`, профильный smoke по измененному модулю; для склада/снабжения дополнительно `npm run smoke:supply-chain`, для workflow-накладных `npm run smoke:workflow-invoice`, для действий пользователей `npm run smoke:activity-log`, для сметных материалов `npm run check:smeta`.

### 2.2. Документы поставщиков, сверка и исторические накладные

Структура связи:

`warehouse_invoices -> suppliers/supplier_aliases -> supplier_invoices -> accounting supplier documents -> supplier cabinet -> future notifications`

Границы ответственности:

- `warehouse_invoices` - факт прихода на склад и фото/позиции накладной.
- `suppliers` + `supplier_aliases` - единая карточка поставщика и все варианты названия/ИНН из документов.
- `supplier_invoices` - бухгалтерская первичка поставщика: счет, УПД, долг, статус проверки и связь с приходной накладной.
- `Бухгалтерия / Документы от поставщиков` - рабочая сверка бухгалтера по поставщикам: все накладные, счета, фото, позиции, оплаты и расхождения.
- `Кабинет поставщика / Документы` - внешний вид той же цепочки для поставщика после регистрации или привязки аккаунта.
- `messenger_accounts` - будущий канал MAX-уведомлений, если поставщик авторизовался или привязал MAX.

Правило расчетов и связей поставщика:

- Единая карточка поставщика считается не только по одному `suppliers.id`, а по группе связанных поставщиков: явный `supplier_id`, `user_id`, email, телефон, ИНН, ОГРН и явные `supplier_aliases`. Нормализованное название без сильных реквизитов является поводом для `Нужно уточнение` или ручной связки, а не автоматическим основанием для объединения.
- Физические дубли поставщика не удаляются обычным интерфейсом. UI, кабинет поставщика и backend-scope для роли `поставщик` должны читать все связанные `supplier_id`, а не только первый найденный ID.
- Если зарегистрированный поставщик вошел как физлицо или не по правильному инвайту и не видит компанию, это не исправлять переименованием карточки. Директор должен явно связать пользователя с карточкой поставщика через `suppliers/{id}/link-user`; endpoint проставляет `suppliers.user_id`, пишет alias и переносит явную привязку с ошибочной карточки, не проводя документы и не удаляя дубли.
- Источник поставщика хранить в карточке отдельными полями `source_type/source_detail`: `invite_link` - поставщик сам пришел по ссылке, `site`/`crm` - заявка с сайта или CRM, `warehouse_invoice`/`max_invoice` - поставщик создан из приходной накладной, `manual` - добавлен вручную, `linked_account` - пользователь был привязан к существующей карточке. Источник нужен для управления базой поставщиков и не является основанием для оплаты или складского движения.
- Если накладная пришла раньше регистрации поставщика, система создает или связывает черновую карточку поставщика по реквизитам документа. После регистрации по email/ИНН/ОГРН/телефону или после ручной связки дубля поставщик должен увидеть старые накладные и счета своей группы.
- `Счета` считаются из `supplier_invoices` по связанным `supplier_id` или по связанному названию/алиасу поставщика. Сумма счета = `amount`, оплачено = `paid_amount`, долг = `max(0, amount - paid_amount)`.
- `supplier_invoices` создается идемпотентно: повторный счет/УПД с тем же `company_id + invoice_number + invoice_date + project_name` и совпадением по цепочке КП/заявки, группе поставщика или сумме не вставляет вторую бухгалтерскую строку. Нужно вернуть существующий `id`, дозаполнить связи `offer_id/request_id/warehouse_invoice_id` и связать новую складскую накладную через `warehouse_invoices.supplier_invoice_id`, даже если OCR распознал поставщика другим названием.
- Старые дубли `supplier_invoices` чистить только через управляемую сверку `/supplier-documents/dedupe`: сначала dry-run показывает группы, основной документ и строки на аннулирование; apply разрешен только для безопасных дублей с одинаковым номером, датой, объектом, компанией, суммой и уверенной связью поставщика/цепочки. Лишняя бухгалтерская строка получает `status='Аннулирован'`, складские накладные перелинковываются на основной счет, складские остатки и оплаты не перепроводятся.
- `Накладные` считаются из `warehouse_invoices`: прямой ключ `supplier_invoice_id` главный; если его нет, связывать через supplier name/aliases/ИНН и объект/дату/номер. Накладная является фактом прихода на склад, но не фактом оплаты.
- `Поставки` считаются из `supply_deliveries` и выбранных КП/заявок, связанных с поставщиком. Поставка показывает статус исполнения заявки, но не должна повторно увеличивать склад, если уже создана складская накладная.
- В карточке поставщика счетчики и сумма должны считать уникальные документы, а не сырые строки: дубли складских накладных, счетов и поставок определяются по номеру/дате/поставщику/объекту, а при пустом номере - по дате, поставщику, объекту, сумме и сигнатуре позиций. Скрытые повторные строки показывать отдельным предупреждением `дублей документов`, чтобы сумма не раздувалась молча.
- Запрос КП должен работать по группе поставщика, а не по одному физическому `suppliers.id`: `selected_suppliers` хранит все связанные id группы, а `supplier_offers` создается один раз на предпочтительную карточку группы, сначала на карточку с привязанным `user_id`/регистрацией. Если КП было отозвано или отклонено, повторный запрос поднимает ту же запись в `Ожидает ответа`, не создает второй активный запрос для дубля.
- `КП` считаются из `supplier_offers`; отозванные КП не исчезают из истории, а уходят в события `supplier_offer_events`. Выбранное КП может создать счет поставщика и цепочку поставки.
- `Каталог` считается из прайсов/каталога поставщика по связанным `supplier_id`; он нужен для выбора поставщика и цен, но сам не создает заявку, счет, склад или расход.
- Факт денег не выводить из счета или накладной. Оплата поставщику должна одновременно обновить `supplier_invoices.paid_amount/status` и записать движение в `project_payments` по объекту/пакету с понятной заметкой `Оплата счёта ...`. Без `project_payments` это только обязательство, а не фактический расход.
- Бухгалтерская сверка должна показывать расхождения: счет без накладной, накладная без счета, оплата без первички, долг по счету, поставщик без уверенной связи, фото/позиции отсутствуют, сумма/НДС не сходятся.
- Любая неуверенная связь поставщика или старой накладной должна получать статус `Нужно уточнение`/`needs_review`, а не молча склеиваться с чужим поставщиком.

Правило для каждого блока: не менять бизнес-логику, только переносить код; после переноса выполнять `npm run build`, `PYTHONPYCACHEPREFIX=/tmp/stroyka-pycache python3 -m py_compile backend/main.py`, `git diff --check`. Серверный deploy делать только после commit/push.

Post-refactor gate:

- После каждого блока рефакторинга обязательно проверить, что приложение не сломалось, до перехода к следующему блоку.
- Минимум локально: `npm run build`, `PYTHONPYCACHEPREFIX=/tmp/stroyka-pycache python3 -m py_compile backend/main.py`, `git diff --check`.
- Если блок затрагивает frontend-экраны, дополнительно открыть затронутый экран и один смежный экран старой цепочки: например `Проекты -> объект`, `Сметы -> смета`, `Склад -> накладная`, `ЖПР -> запись`, `Сайт -> /`; для production после deploy запускать `npm run smoke:browser-prod`, чтобы поймать белый/черный экран и React runtime-ошибки, которые обычный HTTP smoke не видит.
- Для проверки падений после входа запускать browser-smoke с тестовым пользователем: `BROWSER_SMOKE_EMAIL='...' BROWSER_SMOKE_PASSWORD="$PASS" npm run smoke:browser-prod`. Если у пользователя 2FA, дополнительно передать `BROWSER_SMOKE_2FA_CODE` или использовать тестовую роль без 2FA.
- Если блок затрагивает backend/API, дополнительно запустить профильный smoke: `smoke:prod` для базовой доступности, `smoke:supply-chain` для склада/снабжения, `smoke:work-doc-chain` для ЖПР/актов, `smoke:estimate-norms` для смет/норм, `smoke:role-matrix` для прав ролей.
- Если проверка падает или экран показывает белый/черный экран, следующий рефакторинг запрещен: сначала исправить поломку, повторить проверки и только потом продолжать распил.
- После серверного deploy обязательно выполнить `npm run smoke:prod`; если есть доступ к тестовому пользователю без 2FA или с TOTP, прогнать protected smoke, а не только публичные маршруты.
- Каждый отчет по рефакторингу должен содержать: сколько строк вынесено, какие файлы затронуты, какие проверки прошли, что не проверялось и почему.

- Отрицательные материалы сметы не участвуют в заявках, остатках и списаниях.
