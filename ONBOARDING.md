## Контекст проекта

**stroyka-app** — система учёта для строительной компании «СтройКа». Главная цепочка: смета → выполнение → журнал работ/исполнительная документация → КС-2/КС-3 → финансы. Также есть склад, снабжение, персонал и AI-помощник.

**Размер бизнеса:** ~40 сотрудников, разные бригады по специализациям, работа с ТД / ГПХ / Самозанятыми / ИП.

## Мобильная стратегия

- До стабилизации продакшен-ядра делаем **мобильную веб-версию/PWA**, а не отдельное приложение. Причина: сейчас важнее, чтобы одна логика сметы, склада, журналов, актов и ролей работала одинаково на телефоне и компьютере.
- Полноценное мобильное приложение нужно планировать **после продакшен-стабилизации**: когда роли, права доступа, документы, снабжение, сметы, материалы, фото и offline-сценарии уже не будут постоянно меняться.
- Первый кандидат на отдельное приложение — **полевой контур**: мастер, электрик, слаботочник, кладовщик, прораб. Им нужны быстрые действия с телефона: фото, журнал работ, кабельный журнал, входной контроль, выдача/списание материалов, замечания, QR-документы, работа при слабом интернете.
- Рекомендуемый стек будущего приложения: **Expo / React Native** с тем же backend API. Не плодить отдельную бизнес-логику в приложении: backend остаётся единым источником правды.
- Перед стартом мобильного приложения обязательно подготовить API: стабильные endpoint по ролям, короткие payload, нормальные ошибки `401/403/422`, загрузка фото в объектное хранилище, push-уведомления, refresh-токены, offline-очередь действий.
- MVP мобильного приложения:
  1. вход по логину/паролю и сохранённая сессия;
  2. список своих объектов;
  3. журнал работ с фото и материалами;
  4. кабельный журнал для электриков/слаботочников;
  5. входной контроль/приёмка материалов;
  6. предписания и замечания с фото;
  7. уведомления по назначенным задачам;
  8. offline-черновики с отправкой после появления связи.
- Заказчик и технадзор на первом этапе могут оставаться в мобильной веб-версии: им важнее просмотр, печать/подпись документов и замечания, а не тяжёлый полевой ввод.

## Укрепление ядра — 2026-05-28

- Пароли больше не должны возвращаться из `/login` и `/register`.
- Пользователей в продакшене не удаляем физически: `DELETE /users/{id}` должен отключать доступ (`active=false`), сохраняя историю журналов, актов и задач. Отключённый пользователь не может войти.
- Backend проверяет `active` не только при `/login`, но и при каждом запросе с токеном: отключение аккаунта сразу гасит уже выданную сессию.
- Пароль пользователю меняется через редактирование в разделе `Пользователи`: поле `Новый пароль` отправляется в существующий `PUT /users/{id}`. Отдельный endpoint сброса для директора пока не нужен.
- Перед реальным запуском сделать prelaunch-чистку аккаунтов: после свежего backup оставить активным только директора/владельца, отключить все тестовые роли (`active=false`), сменить пароль директора, затем создать реальных прорабов, мастеров, бухгалтерию, технадзор и поставщиков заново с правильными объектами.
- Новые пароли пишутся как `pbkdf2_sha256`; старые plaintext-пароли поддерживаются для входа и автоматически хешируются при успешном логине.
- `/password-reset-request` больше не возвращает `_devCode`, кроме режима `SHOW_RESET_CODE=true` в `.env`.
- `VK_TOKEN` вынесен из кода в `.env`; старый токен нужно перевыпустить, потому что он уже попадал в репозиторий.
- CORS больше не `*`; по умолчанию разрешены домены `stroyka26.pro`, IP сервера и localhost.
- Починен `POST /projects`: теперь передаются `floors` и `liters`, создание объекта не должно падать из-за mismatch placeholders.
- Следующий слой безопасности: добавить токен авторизации на frontend-запросы и backend-проверку ролей по всем критичным API.

## Укрепление API — 2026-05-29

- `/login` и `/register` возвращают `authToken`; frontend кладёт его в `localStorage.authToken`.
- В `src/App.js` установлен общий wrapper над `fetch`: все API-запросы автоматически получают `Authorization: Bearer ...`.
- При выходе `authToken` удаляется.
- Backend умеет проверять токен через HMAC (`AUTH_SECRET` из `.env`, fallback от `DB_PASSWORD`; лучше задать отдельный `AUTH_SECRET` на проде).
- Закрыты серверной проверкой ролей основные опасные записи:
  - объекты/клиенты: создание/удаление только руководство, редактирование объекта также прораб/ГИП/сметчик;
  - пользователи: только руководство;
  - склад/материалы/накладные/передачи: складские роли;
  - снабжение/КП/поставки/претензии: роли снабжения;
  - счета поставщиков и поступления денег: бухгалтерия/руководство;
  - системный кабинет: только `system_owner`;
  - загрузка файлов требует авторизации.
- Это первый слой защиты. Чтение части справочников пока открыто, потому что текущий `loadAll()` грузит данные централизованно для всех ролей; следующий шаг — разнести чтение по ролям и проектам.

## 🔗 Единая цепочка без дублей

Главное правило системы: **один факт выполнения имеет один источник — смета или утверждённое изменение к смете.** Журнал, АОСР, КС-2/КС-3 и расчёт с бригадой только отражают этот факт, но не создают второй параллельный объём.

### Как должно идти
```
ДОГОВОРНАЯ СМЕТА
   ├─→ мастер/прораб заполняет «Сделано»
   │      ├─→ ЖПР создаётся автоматически
   │      ├─→ если позиция 🔒, создаётся печатная форма АОСР
   │      ├─→ КС-2/КС-3 берут объём и цену заказчику из сметы
   │      └─→ расчёт с бригадой берёт тот же объём, но внутреннюю цену бригады
   └─→ если объём/состав не совпал
          └─→ создаётся «Изменение к смете» и проходит согласование
```

### Сметы в этом процессе
- **Договорная смета** — базовая версия договора с заказчиком. Её нельзя переписывать задним числом ради факта.
- **Активная смета** — текущая рабочая версия, из неё идут «Сделано», ЖПР, АОСР, КС-2/КС-3 и прогресс.
- **Архивные версии** — история. Они нужны для сравнения и доказательства, но не должны участвовать в текущих расчётах.
- **Уточнённая смета** — новая версия после пересогласования. Перед включением в расчёты её нужно сверить с уже утверждёнными изменениями, чтобы не задвоить объёмы.

### Пакеты работ и материальный контроль — план внедрения
На одном объекте может быть несколько активных смет: `Общестрой`, `Отопление`, `Электрика`, `Слаботочка`, `Вентиляция` и т.д. Активность должна работать **внутри пакета**, а не по всему объекту: новая активная «Электрика» не должна архивировать активную «Общестрой».

Этапы:
1. **Пакеты смет** — добавить `workPackage` в сметы, группировать версии по объекту + типу сметы + пакету, прогресс/КС/кабинет заказчика считать суммой всех активных пакетов.
2. **Сравнение смет** — при загрузке новой версии показывать отдельный документ разницы: добавлено, исключено, изменён объём/цена. Документ должен печататься.
3. **Многократные версии** — хранить цепочку версий по каждому пакету: v1, v2, v3, v4. Активная только одна на пакет, архив не участвует в расчётах.
4. **Потребность материалов** — собирать план материалов из всех активных смет объекта без сумм для заказчика/технадзора и с суммами только для внутренних ролей. Важно: в материальный план попадают только строки сметы типа `Материал`, а не любая работа с материальной ценой.
5. **Поставки и перемещения** — приход может идти на общий склад, потом перемещаться на объект. Сравнение «по смете / поставлено / перемещено / списано / остаток» должно учитывать оба пути.
6. **Списание материалов** — закрывать материал по факту работ/пакету, чтобы видеть перерасход и недопоставку.
7. **Онлайн-проверка ролей** — после каждого этапа прогонять цепочку на сервере: админ/прораб/мастер/заказчик/технадзор/склад.

### План 1 — проект, обмеры и постоянный ИИ-контроль объекта

Цель: сделать единый контур `проект/фактический обмер → помещения → смета → факт работ → материалы → поручения`, чтобы система заранее показывала, чего не хватает, где смета расходится с обмером и кто должен это исправить.

Базовые принципы:
- Если есть проект — загружаем PDF/планы/экспликации/ведомости окон и дверей во вкладку `Проект / Обмеры`; AI создаёт черновик помещений с номерами, названиями, площадями, окнами, дверями и откосами.
- Если проекта нет — источник истины `Фактический обмер`: помещения создаются вручную или из фото/скана рукописных записей через OCR + AI-разбор.
- Любой AI-разбор имеет статус `Черновик ИИ → На проверке → Принято`; AI не меняет смету, обмеры, списание, акты и оплаты без подтверждения человека.
- У каждого помещения хранится источник: `Проект`, `Фактический ручной`, `Фактический из фото`, `Уточнённый`.
- Окна и двери обязательно вычитаются из общей площади стен: `стены к отделке = wallArea - окна - двери`. Оконные и дверные откосы считаются отдельными объёмами и не смешиваются со стенами.
- Демонтажные работы ведутся отдельным слоем: демонтаж стен/пола/потолка/окон/дверей привязан к помещению и элементу, но не считается монтажной отделкой.

Что нужно добавить в данные:
- `project_measurements` для загрузки проекта, экспликаций, ведомостей, фото обмеров и статуса разбора.
- Расширение `rooms`: номер помещения, тип, этаж, литер, источник обмера, статус проверки, периметр, высота, общие/чистые стены.
- Элементы помещения: окна, двери, проёмы, откосы, подоконники, ниши/колонны/короба, линейные элементы, инженерные точки.
- Поле у строки сметы/работы `measurementBasis`: `wall_net_area`, `wall_gross_area`, `floor_area`, `ceiling_area`, `window_reveals`, `door_reveals`, `window_count`, `door_count`, `linear`, `point`, `demolition`, `manual`.
- `ai_findings` — замечания AI: что не сходится, важность, ссылка на объект/помещение/смету/материал/ЖПР, предложенное действие.
- `ai_tasks` — поручения по замечаниям: кому назначено, срок, статус, кнопка `Принять к исполнению`, история закрытия.

Как должен работать постоянный агент:
- Агент событийный, а не бесконечный чат. Запускается после загрузки проекта/обмера/сметы, изменения помещения, отправки работы, списания материала, появления допработы и ночным прогоном по активным объектам.
- 70% проверок делаются правилами в коде почти бесплатно: полнота обмеров, вычитание окон/дверей, привязка работ к помещениям, превышение сметы/обмера, материалы по плану/поставкам/выдачам/списанию.
- Модель подключается только где нужен язык/изображения/понимание: PDF проекта, рукописный обмер, классификация строк сметы, объяснение замечаний, подсказки по АОСР и изменениям к смете.
- AI-провайдер должен быть сменным: `YandexGPT / OpenAI / локальная модель`. В настройках компании — лимиты расходов в день/месяц и отключение логирования у провайдера, где это доступно.
- Агент обучается не на стороне модели, а внутри нашей БД: справочник соответствия работ поверхностям, нормы расхода, решения сметчика, типовые названия помещений и материалов.

Где подсвечивать:

- Кабинет мастера: только практичные блокировки и подсказки — выбрать помещение, указать поверхность, превышение по обмеру, нет выданного материала.
- Кабинет роли: снабженец/замдиректора получает материалы, прораб/инженер — обмеры и привязки, сметчик — расхождения сметы, бухгалтер — финансовые риски.

Режимы компании для продажи продукта:

### Конкурентные выводы — что доработать, не распыляя продукт

Позиционирование: не строить “ещё одну ERP обо всём”. Ядро продукта — **контроль потерь подрядчика**: не потерять объём, материал, допработу, документ и ответственное действие. Любая новая функция должна быть привязана минимум к одному из узлов цепочки: `объект → помещение/проект → строка сметы → факт → материал → документ → поручение`.

Что у конкурентов сильнее и как внедрять у нас:
- **Мобильная полевая работа** (`PlanRadar`, `Gectaro`, `МонСтр`): сильны мобильные сценарии, фото, задачи, быстрый доступ с объекта. У нас это должно лечь не отдельным приложением “про всё”, а как PWA/мобильный режим мастера и прораба: принять работу, фото, помещение, обмер, материал, замечание, поручение AI.
- **Задачи на плане/чертеже** (`PlanRadar`, `МонСтр`): сильны метки на планах. У нас это внедрять через вкладку `Проект / Обмеры`: план помещения, точки замечаний, окна/двери/откосы, привязка к комнате и строке сметы. Не делать отдельный “чертёжный модуль” без связи со сметой.
- **Исполнительная документация и нормативная зрелость** (`BuildDocs`, `HARDROLLER`, `МРС`, `АЛТИУС`): сильны формы, реестры, согласования, ЭП/XML. У нас следующий слой — привести ИД к строгому реестру: ЖПР, АОСР, входной контроль, кабельный журнал, предписания, КС, сканы, подписи, статусы, печать; позже ЭП/XML.
- **ERP/снабжение/финансы** (`Gectaro`, `STROICON`, `buildbuild`): сильны склад, заявки, счета, финансы. У нас уже есть широкая цепочка, но нужно уплотнить её вокруг сметы: материал только как `по смете / вне сметы / по изменению`, снабжение из потребности, оплата через документы, без параллельных источников факта.
- **SaaS-упаковка и продажи** (`STROICON`, `Gectaro`, `PlanRadar`): сильны onboarding, тарифы, роли, тестовый период, поддержка. У нас нужно добавить настройки компании, тарифные лимиты, AI-бюджет, демо-объект, пошаговый мастер запуска и понятный режим `Микро / Стандарт / Расширенный`.

Очередь конкурентных доработок:
1. **Собрать интерфейс вокруг объекта**, а не вокруг десятков разрозненных разделов: объект как главный рабочий экран, внутри `Проект/Обмеры`, `Смета`, `Работы`, `Материалы`, `Документы`, `ИИ-контроль`.
2. **Мобильная PWA для поля**: мастер/прораб видит только свои действия, фото, обмер, помещение, материалы, задачи; проверка на телефоне обязательна.
3. **План/чертёж как слой помещений**: загрузка плана, метки замечаний, комнаты, окна/двери, связь с обмером и сметой.
4. **Документальная зрелость**: реестры ИД, версии документов, строгие формы, история подписания, загрузка сканов; ЭП/XML отдельным поздним этапом.
5. **SaaS-готовность**: тарифы, лимиты AI, биллинг, демо-онбординг, файловое хранилище, мониторинг ошибок, резервные копии, политика данных.

Анти-распыление:
- Не добавлять отдельный модуль, если он не создаёт связь с объектом, помещением, сметой, материалом, документом или поручением.
- Не делать AI-кнопки ради AI: AI должен создавать проверяемое замечание, черновик или поручение.
- Не копировать конкурентов по ширине. Наша сила — глубина связки `обмеры ↔ смета ↔ факт ↔ материалы ↔ документы`.

### План Авито + логистика

Зафиксирован как будущий модуль, **не вставлять в проект как отдельную открытую надстройку**. Правильная цель — связать продажи остатков/материалов, входящие лиды и доставку с уже существующими контурами `CRM ↔ склад ↔ снабжение ↔ объект`.

### Изменения к смете
Текущую сущность `unexpected_works` нужно использовать/переименовать как **«Изменения к смете»**, а не как свалку непредвиденных работ.

Типы изменений:
- **Дополнительный объём к строке сметы** — работа есть в смете, но фактический объём больше. Обязательна привязка к строке сметы.
- **Работа вне сметы** — такой работы нет в договорной смете. Создаётся отдельной строкой и отдельным блоком в КС-2/КС-3.
- **Замена решения** — одну работу/материал убрали, другой добавили.
- **Исключение объёма** — часть договорной работы не выполняется и должна уменьшить объём/сумму.

Статусы изменений:
- `Черновик`
- `На согласовании`
- `Утверждено отдельной допработой`
- `Включено в новую смету`
- `Отклонено`

Ключевой запрет: **одно изменение не может одновременно считаться отдельной допработой и входить в новую активную смету.**

### Если факт больше сметы
Пример: в договорной смете `Штукатурка 100 м2`, по факту нужно `200 м2`.

Правильный сценарий:
1. По строке договорной сметы закрываем максимум `100 м2`.
2. Система предлагает оформить превышение `+100 м2` как **«Дополнительный объём к строке сметы»**.
3. Изменение получает ссылку на исходную строку сметы: `estimateId`, `sectionName`, `estimateItemName`, `baseQuantity`, `deltaQuantity`.
4. Заказчик согласовывает или отклоняет.
5. Если утверждено отдельной допработой — КС-2/КС-3 показывают её отдельным блоком.
6. Если позже пришла новая смета, где уже `200 м2`, изменение переводится в `Включено в новую смету` и отдельно больше не считается.

### Если работы вообще нет в смете
Это тип **«Работа вне сметы»**. Она не привязывается к существующей строке, но должна иметь причину, объём, цену, фото/обоснование и согласование. В КС-2/КС-3 идёт отдельным блоком «Работы вне сметы».

### При загрузке новой сметы
Новая смета не должна автоматически затирать старую без следа. При создании/импорте/активации новой активной сметы система делает фоновую сверку:
- сравнивает старую активную смету и новую;
- формирует печатную сопоставительную ведомость: добавлено, исключено, изменён объём/цена, плюс отдельный блок утверждённых изменений к смете;
- сопоставляет уже утверждённые изменения к смете с новой сметой;
- очевидные совпадения сама переводит в `Включено в новую смету`, чтобы КС-2/КС-3 не задвоили объём;
- спорные позиции оставляет отдельной допработой и создаёт одно поручение в `ИИ-контроль`: `Проверить включение допработ`.

Ручная кнопка включения изменений остаётся только как запасной сценарий, когда новую редакцию сметы нужно создать из утверждённых изменений внутри системы. Обычный путь: загрузили/активировали новую смету → система сама сверила → человек проверяет только спорное.

### Документы
- **ЖПР** — создаётся из выполненных объёмов сметы/утверждённых изменений. Это журнал факта, не источник денег.
- **АОСР** — печатная форма к ЖПР по позициям сметы с `hiddenWork=true`. Не финансовый процесс и не обязательный стоп-кран.
- **КС-2** — три блока: работы по договорной смете, дополнительные объёмы к смете, работы вне сметы.
- **КС-3** — итог стоимости по тем же источникам, без дублей.
- **Расчёт с бригадой** — использует объём из сметы/изменений, но цену бригады. Не должен иметь своё независимое «сделано».

### Что уже сделано
- `update_estimate` синхронизирует «Сделано» из сметы в ЖПР, АОСР и `brigade_contract_items.done_quantity`.
- `ks2ItemsFromEstimate`, `showKS2`, `buildKS2Content`, `buildKS3Content` уже считают из сметы заказчика + утверждённых допработ.
- Оплаты бригаде создают `brigade_payment` и связанное движение `project_payments`; АОСР к деньгам не относится.

**Важно для проверки на сервере:** выполнение в расчёте с бригадой обновляется при сохранении сметы. Для старых legacy-расчётов значение подтянется при первом пересохранении сметы объекта или отдельной миграцией.

## Nginx whitelist

В `/etc/nginx/sites-enabled/stroyka` есть `location ~* ^/(endpoints|здесь|перечислены|...)`. **При добавлении нового endpoint** — обязательно проверь покрывает ли его префикс в whitelist, иначе nginx вернёт HTML вместо JSON (классическая ошибка `Unexpected token '<'`). `proxy_read_timeout 300s` для долгих AI запросов, `client_max_body_size 25m` для сканирования фото.

## Особенности БД (ловушки)

- **Таблицы создавались от `postgres`**, потом приложение начало подключаться как `stroyka`. Если backend падает с `must be owner of table`:
  ```sql
  DO $$ DECLARE r RECORD; BEGIN
    FOR r IN SELECT tablename FROM pg_tables WHERE schemaname='public' LOOP
      EXECUTE 'ALTER TABLE public.' || quote_ident(r.tablename) || ' OWNER TO stroyka';
    END LOOP;
    FOR r IN SELECT sequence_name FROM information_schema.sequences WHERE sequence_schema='public' LOOP
      EXECUTE 'ALTER SEQUENCE public.' || quote_ident(r.sequence_name) || ' OWNER TO stroyka';
    END LOOP;
  END $$;
  ```
- **`brigade_contract_items` использует `description`, не `name`!** Также нет колонки `status` — статус вычисляется на лету по `done_quantity` vs `quantity`. Это legacy таблица созданная вручную раньше моих правок.
- **`work_journal` уже имеет колонку `description`** (не `name`).
- При ALTER TABLE моих новых колонок — использовать `ADD COLUMN IF NOT EXISTS` чтобы не сломать существующее.

## 🛒 План снабжения (зафиксирован, ждёт реализации)

Полная цепочка из 15 шагов от потребности до закрытия в Бухгалтерии. **Снабженец убран из цепочки — заявку обрабатывает директор+AI.**

### Цепочка:
1. Мастер/Субподряд создаёт запрос материала
2. Прораб подтверждает (или сам создаёт заявку)
3. Директор утверждает (с предупреждением о превышении бюджета)
4. AI подсказывает остатки на складе («есть 30 из 50, закупить 20?»)
5. AI замечает похожие заявки от разных объектов → предлагает объединить
6. AI предлагает поставщиков по категории материала
7. Директор подтверждает кому отправлять (1 клик)
8. Email + кабинет поставщика: RFQ с напоминаниями (2-3 дн → просрочка 5 дн → авто-закрытие 10 дн)
9. Поставщик присылает КП (цена, срок, **условия оплаты**, ±НДС, PDF)
10. AI сравнивает КП по цене и условиям оплаты → рекомендует лучшее
11. Директор выбирает финального (может сменить выбор AI)
12. Поставщик присылает счёт-фактуру
13. Бухгалтер оплачивает по условиям (предоплата/50_50/постоплата)
14. Поставщик «🚚 Отправлено»
15. Прораб скан накладной + AI сверка с КП (недопоставка/брак) → качество ✅/⚠️/❌/🔄 → акт претензии + рейтинг

### Архитектура UI:
- **Новый раздел «🛒 Снабжение»** в сайдбаре (5 вкладок: Заявки / КП / Поставки / Закрывающие / Каталоги)
- Счета поставщиков **переезжают** туда из Бухгалтерии
- На дашборде директора 3 кликабельные карточки: КП на согласование / Заявки / Счета к оплате

### Включено:
- Дробление КП по позициям (заявка с 5 материалами — разным поставщикам)
- Платёжное поручение PDF (генерация при оплате счёта)
- Закрывающие документы (УПД, акты сверки от поставщиков)
- Каталог поставщика (Excel + ручной ввод)
- История цен на материал (видна при создании заявки)
- Шаблоны заявок (повторяющиеся наборы)
- AI распознаёт счёт/УПД (OCR + сверка)
- Авторейтинг поставщиков (брак — минус)
- Регистрация поставщиков: самореги по форме + инвайт-код от руководства

### Отложено:
- Интеграция с 1С Бухгалтерия
- Календарь поставок
- Email (SMTP) — пока MVP через кабинет поставщика

### План разбит на 5 этапов (~10-12 часов):
- **Сн.1** Заявки + утверждение + AI остатки склада (~2 ч)
- **Сн.2** AI выбор поставщиков + кабинет поставщика с КП (~3 ч)

**АОСР** — это печатная форма исполнительной документации по СНиП 12-01-2004, а не отдельный финансовый процесс. Используется для работ, которые закрываются другими конструкциями (армирование под бетон, гидроизоляция под облицовку, утепление под отделку и т.п.). Сценарий: смета отмечает позицию `hiddenWork=true` → при заполнении «Сделано» backend создаёт запись ЖПР и форму АОСР → прораб проверяет реквизиты → печать для сдачи заказчику. Подписи в системе можно вести для удобства, но они не обязательный стоп-кран.

**Автоконтроль сметы по нормам материалов** — не плодить кнопки в каждой строке. Правильный сценарий: смету загрузили/создали/активировали/сохранили → система сама прогоняет активную смету заказчика по справочнику норм → если есть несостыковки (`Нет материала в смете`, `Материал без количества`, `Материал без работы`, `Нет нормы`), создаётся одно поручение в `ИИ-контроль`: `Проверить смету: объект — N замеч.`. Если в системе есть сметчик, задача назначается роли `сметчик`, иначе директору. Backend не создаёт дубль по маркеру `ESTIMATE_NORM_REVIEW:<estimateId>`.
Поручения без `findingId` показываются отдельным блоком в `Объект → ИИ-контроль`; кнопка `Открыть проверку сметы` ведёт в `Сметы → Нормы материалов` и выставляет нужный объект. При последующих сохранениях сметы задача обновляет количество замечаний, а когда замечаний не осталось — закрывается автоматически.

**Автосверка новой сметы со старой** — если создаётся/импортируется/генерируется ИИ или активируется новая смета в той же группе (`объект + тип + пакет`), система сравнивает её с предыдущей активной сметой и создаёт одно поручение `Сверить разницу смет`. Кнопка `Открыть разницу смет` открывает печатный документ с изменёнными, добавленными и исключёнными позициями. Это нужно перед включением утверждённых изменений в новую смету, чтобы не задвоить объёмы и КС.

## Workflow

- Worktree `claude/cool-bohr-af2b82`, fast-forward merge в main, push в origin
- Деплой — пользователь сам через `bash deploy.sh` на сервере
- **Не коммитить без явного «давай коммить»** — кроме случаев когда явно идёт работа над фичей
- При проблеме спрашивать что видит на сайте + логи `journalctl -u stroyka -f`
- **Скриншоты** — пользователь умеет (Cmd+Shift+4 на маке)
- Часто использовать **AskUserQuestion** для развилок где нужно его мнение
- ⚠️ **При обновлении nginx whitelist** — бэкапы класть в `/etc/nginx/backups/`, НЕ в `/etc/nginx/sites-enabled/` (иначе nginx подхватит их и будут conflicting server warnings)
- Бэкап БД работает через cron 03:00 ежедневно — `/root/backup-stroyka.sh` → `/var/backups/stroyka/`
