## Суть проекта

`stroyka-app` - система управления строительной компанией.

Главная цепочка:

`объект -> обмеры/проект -> сметы -> пакеты работ -> исполнители -> ЖПР/журналы -> АОСР/КС -> материалы -> финансы -> бухгалтерия`

Смета заказчика должна быть источником правды для работ, материалов, КС и экономики. Для объектов без ГРАНД-сметы допускается отдельный режим работы по внутреннему прайсу.

## Целевая цепочка исполнения

### 1. Объект и сметы

- У объекта может быть несколько сметных пакетов: `общестрой`, `отделка`, `электрика`, `слаботочка`, `отопление`, `вентиляция`, `ВК/канализация`, `благоустройство` и т.д.
- Базовая единица исполнения - не вся смета и не старый наряд, а `пакет работ` внутри объекта.
- Заказчик через API и кабинет видит только активные сметы типа `Заказчик` по своему объекту. Внутренние, материальные, исполнительские и черновые сметы заказчику не отдавать.
- Материальные/ресурсные строки сметы с количеством `< 0` или суммой `< 0` считать корректировкой/исключением: учитывать в общей сумме сметы, но не включать в графу `Материалы`, заявки, потребность, контроль остатков и списание. Строки с количеством `0` не формируют заявки/списание, пока нет реального количества.
- Покрытие нормы материала считать только по положительным ресурсным строкам. Если по норме нужно больше, чем есть положительного материала в смете, статус должен быть `Нехватка материала по норме`; отрицательные строки не закрывают норму.
- При списании через ЖПР норму материала должен подтверждать сервер. Нельзя доверять `normQuantity`, который пришел только с фронта: если сервер не нашел правило нормы, списание по такой "норме" не принимать.
- Исполнитель получает доступ к связке `объект + пакет`, а не ко всему объекту.
- Мастер, бригада или субподрядчик не должны видеть чужие сметы, чужие пакеты и чужие расценки.
- Цена заказчика и цена исполнения должны храниться отдельно:
  - цена заказчика - для КС, финансов и маржи;
  - цена исполнения - для мастера, субподрядчика, бухгалтера и выплат.

### 2. Помещения и зоны

- Если объект ведется по помещениям, факт работы закрывается по связке `объект + пакет + помещение + строка сметы + исполнитель`.
- Помещение выбирается из справочника объекта, не вводится свободным текстом.
- Для фасада, кровли, наружных сетей и благоустройства вместо помещения используется зона/участок/фронт работ.
- Окна, двери и откосы должны участвовать в обмерах: их нельзя просто выкинуть из расчета, потому что они могут быть отдельными работами в смете.
- Система должна не давать закрыть в помещении больше объема, чем допускают обмеры и уже подтвержденное выполнение.

### 3. Факт выполнения

- Исполнитель вводит объем один раз.
- Из этого факта система автоматически ведет:
  - ЖПР;
  - профильные журналы;
  - историю помещения/зоны;
  - кандидатов в АОСР;
  - накопление выполненных объемов;
  - основу для актов исполнителя;
  - управленческую аналитику.
- Одна и та же работа в одном помещении не должна закрываться двумя исполнителями без явного разбиения по объему, этапу, зоне или операции.

### 4. Материалы

Материал должен проходить полный путь:

`смета/норма -> потребность -> заявка -> приход -> перемещение на объект -> выдача исполнителю -> подпись получения -> списание через ЖПР`

Правила:

- Нормы расхода не заменяют физическую выдачу материала.
- Исполнитель не должен списывать материал, который ему не выдали.
- Выдача материала исполнителю должна быть привязана к объекту, пакету и исполнителю.
- Выдача материала разрешена только конкретному активному пользователю с ролью `мастер`, `бригадир` или `субподрядчик`. Абстрактной `бригаде` материал не выдавать: если материал получает бригада, в системе должен быть конкретный бригадир/субподрядчик с доступом к объекту и пакету.
- Перед выдачей материала исполнителю сервер обязан повторно проверить сметный контроль по объекту и пакету. Передача со склада объекта в личный остаток исполнителя не считается новой закупкой: общий покрытый объем не меняется, но `вне сметы` и `сверх лимита` должны блокироваться.
- Приход материала на объект должен идти через цепочку `заявка -> КП -> отгрузка -> приемка`: приемка автоматически создает накладную, историю снабжения, входной контроль и остаток материала на объекте. Ручная приходная накладная на объект запрещена, чтобы не задвоить склад и бухгалтерию.
- В заявках снабжения исполнители (`мастер`, `бригадир`, `субподрядчик`) видят контроль объёмов/лимитов по смете, но не видят сметные деньги заказчика (`plannedSum`, `plannedWorkSum`). Денежный контроль остаётся у директора, замдиректора, снабжения, склада и бухгалтерии.
- В балансе списания материалов учитывать только действующие записи ЖПР. `Отклонено` и `Аннулировано` не должны съедать личный остаток исполнителя и не должны блокировать повторную работу по помещению.
- Материалы нужно сверять со сметой, накладными, перемещениями, выдачей, списанием и остатком.
- Себестоимость материалов объекта считать по фактическому приходу (`накладные/поставки/история снабжения`), а не по текущему остатку `materials`. Остаток нужен для складского контроля; после выдачи/списания он уменьшается, но расход объекта уже состоялся и не должен исчезать из экономики.
- Если материал из накладной не выделен отдельной ресурсной строкой сметы, но относится к укрупненной работе сметы (`вентилируемый фасад`, `кровля`, `инженерная система` и т.п.), его нужно привязать к этой работе как `Комплектацию укрупненной работы`. Такая строка не считается `вне сметы`, не блокирует приход/заявку, но не считается обычным ресурсом сметы и не закрывает отдельную сметную потребность. Она обязана оставаться видимой в фактических затратах, поставщике, объекте, пакете работ и бухгалтерии.
- Если материал не найден в смете и не привязан к укрупненной работе, он считается `вне сметы` и требует проверки/доп. сметы/управленческого решения.

### 5. Бухгалтерия и договоры

- Бухгалтер должен получать не разрозненные данные, а готовый реестр закрытия по исполнителю, объекту и пакету.
- Один субподрядчик может работать на нескольких объектах: карточка исполнителя общая, договоры/акты/платежи привязаны к объектам.
- Месячное закрытие должно собираться из подтвержденных объемов ЖПР и внутренних расценок.
- Удержание 5 процентов на 1 год должно отражаться в договоре, акте, оплате и остатке.
- `Мои траты`: `own_expenses` хранит реестр возмещения сотруднику, `expenses` хранит фактический финансовый расход. При создании траты из веба или Telegram backend автоматически создает связанную строку в `expenses`; при статусе `Отклонено` связанный расход удаляется из экономики объекта; при возврате в рабочий/возмещенный статус расход создается снова. Telegram-доступ идет только через `X-Telegram-Bot-Token` и зарегистрированный Telegram ID/chat ID сотрудника.
- Бухгалтерская сводка: факт оплаченных денег считать из `project_payments` плюс подотчетные, а счета поставщиков и акты исполнителей использовать как основание/долг. Нельзя повторно прибавлять `supplier_invoices.paid_amount` или `interim_acts.paid_amount`, если оплата уже отражена в платежном журнале.

### 6. График руководителя

График объекта должен строиться автоматически из:

- активных смет и пакетов;
- обмеров помещений/зон;
- назначенных исполнителей;
- факта выполнения по ЖПР;
- АОСР и этапов контроля;
- прихода, перемещения и выдачи материалов;
- дедлайнов объекта и этапов.

Руководитель должен видеть план, факт, отставания, блокировки по материалам, проблемные помещения и зависимые этапы.

## Жесткие правила

- Меню роли и backend API должны совпадать. Если раздел скрыт в меню, прямой API-запрос тоже должен быть закрыт. Прайс-листы доступны только ролям, которым они нужны по работе; исполнители видят прайс только на объектах без активной сметы заказчика.
- Не удалять и не архивировать объект обычной кнопкой, `DELETE /projects` или обычным `PUT /projects`.
- Объект закрывается только отдельной процедурой директора: бэкап, проверка документов, проверка смет/финансов, затем явное закрытие.
- Не переводить старую активную смету в архив автоматически при загрузке или активации новой версии. Старая версия должна оставаться видимой как черновик/история группы.
- Безвозвратное удаление сметы разрешено только директору и только если смета не связана с ЖПР, АОСР, договорами, материалами, заявками, поставками и накладными. Для остальных ролей использовать `Снять активность` или `Черновик`.
- Сотрудников не удалять физически из production. Отключение сотрудника = статус `Уволен` + выключенный пользователь, чтобы договоры, акты и история не теряли ссылку на исполнителя.
- `scripts/prelaunch-cleanup.sh` не должен чистить production. Разрешен только `DRY_RUN=1`.
- Тестовые артефакты `CODEX QA` убирать только через `scripts/manage-test-artifacts.py`: сначала dry-run, затем явное `--apply --confirm CODEX_QA`. Скрипт не удаляет данные, а переводит тестовые сметы в `Черновик` и отменяет тестовые заявки.
- Перед любым действием с боевой БД создать бэкап и понимать, как откатиться.
- Не делать общий коэффициент на всю смету. Мастерам платят по конкретной строке и фактическому объему.
- Старые ручные наряды не развивать. Новые работы идут от сметы или прайсового объекта.
- Удаление подписанных передач материалов запрещено. Нужен отдельный возврат.
- Если список длиннее 10 элементов, нужен поиск/фильтр.
- Nginx backup класть только в `/etc/nginx/backups/`, не в `/etc/nginx/sites-enabled/`.
- `outputs/`, `tmp/`, локальные тестовые выгрузки и секреты не коммитить.

- Добавлен и пройден smoke `npm run smoke:staff-access`: директор создает/обновляет карточку сотрудника в `Персонале`, карточка выдает системный доступ, сотрудник входит по паролю, видит только назначенный объект/пакет, не может записать ЖПР в чужой пакет, смена пароля через карточку блокирует старый пароль. Smoke по умолчанию не должен выбирать пакеты `CODEX QA`; если они всплывают в живом объекте, перед продакшеном запускать `scripts/manage-test-artifacts.py`.
- Добавлен smoke `npm run smoke:role-matrix`: создает/обновляет временных пользователей `role-matrix-*`, проверяет вход и ключевые API для директора, замдиректора, главного инженера, прораба, мастера, субподрядчика, сметчика, снабженца, кладовщика, бухгалтера, технадзора и заказчика. Скрипт не заменяет ручной просмотр кабинета, но быстро ловит сломанный вход и грубое расхождение API-доступов.
- Блок автоматических проверок ролей закрыт единым запуском `npm run smoke:work-roles`: `role-package`, `staff-access` и `role-matrix` идут подряд после preflight-входа.
- Прогнать реальные входы и рабочие цепочки: директор, замы, главный инженер, прораб, мастер, сметчик, снабжение, кладовщик, бухгалтер, технадзор, заказчик.
- После следующих изменений проверять, что матрица меню и API-доступов не разошлась.
- Проверить кабинет прораба на живом объекте: видит основные сметы своего объекта и выполнение по помещениям/зонам, но не видит чужие объекты.
- Проверить карточку исполнителя на живом сценарии: роль доступа, специализация, тип исполнителя, доступные объекты, доступные пакеты, схема оплаты.
- Кабинет владельца платформы проверить отдельно: компании, тарифы, пользователи, статус системы, изоляция данных.
- Заказчик и технадзор не обязаны подписывать все в системе, но должны иметь просмотр, печать и замечания.

### F. Договоры и бухгалтерия

- Шаблон договора для самозанятого уже есть: НПД, чек `Мой налог`, отсутствие трудовых отношений, гарантийное удержание 5% на 1 год.
- Реквизиты контрагента уже подставляются в договор из карточки сотрудника/профиля исполнителя и реквизитов компании.
- У бухгалтера уже есть папка по исполнителю с разбивкой по объектам, типам договоров, начислению, удержанию 5%, оплатам и остаткам.
- Реестр месячного закрытия уже строится из подтвержденных объемов ЖПР; можно фиксировать акт и печатать реестр.
- Осталось проверить на живом объекте, что акт, удержание, авансы, оплаты и остаток корректно сходятся после реальной выплаты.
- На проде прогнать бухгалтерскую цепочку на живом объекте: счет поставщика -> оплата -> платежный журнал; акт исполнителя -> оплата -> платежный журнал; свои траты -> возмещение -> платежный журнал. Проверить, что факт расходов не задваивается.
- Автоматический smoke подтвердил API бухгалтерских и расходных разделов без новых ошибок; ручной сценарий выше остается обязательным перед продакшеном.

### G. Продакшен-готовность

- Sentry или аналог для ошибок frontend/backend.
- Добавлен smoke `npm run smoke:password-reset`: временный директор получает код восстановления через публичный endpoint, код проверяется через БД, пароль меняется, старый пароль блокируется, новый пароль входит, задача восстановления закрывается. Если SMTP настроен, smoke дополнительно показывает `emailSent=true`.
- 2FA для директора, замов и бухгалтера.
- Миграция старых локальных `/uploads` в S3.
- PDF-экспорт через серверную библиотеку вместо HTML-печати.
- PWA push или Telegram для поручений, предписаний, проверок, поставок и перерасхода.
- Полноценное мобильное приложение делать после стабилизации PWA и API.
- Продолжить рефакторинг `src/App.js` и `backend/main.py`, не смешивая рефакторинг с изменением бизнес-логики.

## Проверка перед закрытием пункта

Backend:

```bash
PYTHONPYCACHEPREFIX=/tmp/stroyka-pycache python3 -m py_compile backend/main.py
```

Frontend:

```bash
npm run build
```

Production smoke:

```bash
SMOKE_EMAIL='admin@stroyka.ru' SMOKE_PASSWORD='***' npm run smoke:prod
```

Data guard после действий с БД:

```bash
npm run smoke:data-guard
```

Если нужно проверить другой объект или другой минимальный набор смет:

```bash
DATA_GUARD_PROJECT='Название объекта' DATA_GUARD_MIN_ACTIVE_CUSTOMER_ESTIMATES=1 npm run smoke:data-guard
```

Core chains smoke после крупных правок:

```bash
SMOKE_EMAIL='admin@stroyka.ru' SMOKE_PASSWORD='***' npm run smoke:core-chains
```

Скрипт последовательно прогоняет базовый production smoke, доступы персонала, ограничения ролей по пакетам, ЖПР/АОСР/акт/оплату, снабжение/приход материалов и веб-цепочку «Мои траты».
Перед запуском цепочек скрипт делает один preflight-вход. Если пароль неверный или аккаунт уже временно заблокирован, остальные цепочки не запускаются, чтобы не набрать 5 неверных попыток подряд.

Optional chains smoke:

```bash
npm run smoke:optional-chains
```

Без Telegram-токена прогонит только восстановление пароля. С Telegram:

```bash
SMOKE_EMAIL='admin@stroyka.ru' SMOKE_PASSWORD='***' SMOKE_TELEGRAM_BOT_TOKEN='123456:ABC...' npm run smoke:optional-chains
```

Telegram smoke после подключения бота:

```bash
SMOKE_EMAIL='admin@stroyka.ru' SMOKE_PASSWORD='***' SMOKE_TELEGRAM_BOT_TOKEN='123456:ABC...' npm run smoke:prod
```

Ожидаемое поведение `/telegram/own-expenses`: без `X-Telegram-Bot-Token` endpoint должен вернуть `403` или `503`, с правильным токеном и несуществующим Telegram ID должен вернуть `404`. Если возвращает `200/201` без токена — это критичная дыра.

Полная проверка цепочки Telegram-трат:

```bash
SMOKE_EMAIL='admin@stroyka.ru' SMOKE_PASSWORD='***' npm run smoke:telegram-expense
```

Скрипт создает/обновляет временного пользователя `telegram-smoke@stroyka.local`, отправляет тестовую трату через `/telegram/own-expenses`, проверяет появление в `own_expenses`, синхронизацию в `expenses`, затем удаляет тестовую трату и связанный расход.

Полная проверка веб-цепочки «Мои траты»:

```bash
SMOKE_EMAIL='admin@stroyka.ru' SMOKE_PASSWORD='***' npm run smoke:own-expense
```

Скрипт создает/обновляет временного пользователя `own-expense-smoke@stroyka.local`, добавляет трату через веб API `/own-expenses`, проверяет реестр `own_expenses`, синхронный расход в `expenses`, смену статуса на `Возмещено`, затем удаляет тестовую трату и связанный расход.

Полная проверка строительной цепочки ЖПР/АОСР/акт/оплата:

```bash
SMOKE_EMAIL='admin@stroyka.ru' SMOKE_PASSWORD='***' npm run smoke:work-doc-chain
```

Скрипт создает/обновляет временного исполнителя `work-doc-smoke@stroyka.local`, создает тестовый договор и договорную позицию, исполнитель добавляет ЖПР с помещением, директор подтверждает работу, система проверяет автосоздание АОСР, создает акт исполнителя, оплачивает его, проверяет платеж объекта, затем физически удаляет только свои тестовые строки договора, ЖПР, АОСР, акта и платежа.

Проверка цепочки снабжения и материалов:

```bash
SMOKE_EMAIL='admin@stroyka.ru' SMOKE_PASSWORD='***' npm run smoke:supply-chain
```

Скрипт берет положительную ресурсную строку из активной сметы объекта, создает тестового поставщика, заявку, КП, отгрузку и приемку, проверяет автоматическую накладную, остаток материала объекта, историю снабжения и запрет ручной приходной накладной на объект, затем физически удаляет только свои тестовые строки.

Проверка ограничения ролей по объекту и пакету:

```bash
npm run smoke:role-package
```

Скрипт создает/обновляет временных пользователей `role-package-worker@stroyka.local` и `role-package-foreman@stroyka.local`, проверяет выдачу по объекту/пакету и запрет записи ЖПР в чужой пакет.

Быстрая матрица ролей и кабинетов:

```bash
npm run smoke:role-matrix
```

Скрипт создает/обновляет временных пользователей `role-matrix-*`, проверяет вход и ключевые API по ролям. `failures` нужно исправлять сразу; `warnings` показывают спорные/необязательные endpoint-доступы, их разбирать по смыслу роли.

Общий прогон работ и ролей:

```bash
read -s PASS
```

Скрипт последовательно прогоняет `smoke:work-doc-chain`, `smoke:role-package`, `smoke:staff-access` и `smoke:role-matrix`. Использовать после правок ЖПР, актов, карточек сотрудников, ролей, меню или ограничений по пакетам.

Скрипт создает/обновляет временного пользователя `password-reset-smoke@stroyka.local`, запрашивает код восстановления через `/password-reset-request`, читает код из БД, меняет пароль через `/password-reset`, проверяет, что старый пароль больше не работает, новый пароль входит, а системная задача восстановления закрыта.
